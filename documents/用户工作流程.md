# 国家统计局泰兴调查队公文处理系统 V3.0 - 用户工作流程总览

> 本文档从**用户视角**提炼全系统的操作工作流，作为所有业务模块的统一流程参考手册。
> 技术实现细节请参阅对应的设计方案文件；约束与规则请参阅 `实施约束规则.md`。

## 一、公文全生命周期工作流

### 1.1 起草阶段 (DRAFTING)

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
点击「起草新公文」          弹出"新建公文"对话框
  ↓ 必须选择公文文种        调用 GET /api/v1/doc-types 填充下拉（含通用、调研分析等）
  ↓ 填写公文标题（可选）    
  ↓ 点击「确认创建」        POST /api/v1/documents/init
                            → 分配 doc_id、写入 doc_type_id
                            → 状态：DRAFTING(10)
                            → 路由跳转 /workspace/:doc_id

进入工作区后：
  ├─ 侧边栏：勾选知识库节点  → context_kb_ids 写入 useEditorStore
  │          捕获快照版本    → context_snapshot_version 写入 store
  ├─ 侧边栏：选择参考范文    调用 GET /api/v1/exemplars?doc_type_id=xxx
  │  （可选，按文种过滤）    → exemplar_id 写入 useEditorStore
  ├─ Action Bar：显示文种标签（只读）
  └─ 开始在 A4 画板上撰写正文

撰写过程中：
  自动保存（每60秒）         POST /documents/{doc_id}/auto-save
  申请编辑锁（首次输入时）    POST /locks/acquire → 获得 lock_token
  心跳续期（Web Worker）      POST /locks/heartbeat
                              ← next_suggested_heartbeat（动态间隔）
```

### 1.2 AI 润色阶段

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
点击「AI 智能润色」          POST /api/v1/tasks/polish
                            请求体携带：
                              - context_kb_ids
                              - context_snapshot_version（后端校验一致性）
                              - exemplar_id（若已选范文，注入 few-shot）
                            ← 返回 task_id（立即返回，不阻塞）

等待过程中：
  A4 画板显示骨架屏动画      GlobalTaskWatcher 监听 SSE 任务状态

任务完成后：
  右下角通知卡片弹出         "AI 润色已完成，点击查看"
  
任务失败时：
  右下角通知卡片弹出         "AI 润色失败：[错误简述]"
  前端：骨架屏消失，恢复正常编辑状态，允许用户手动重新发起润色

点击「查看详情」（仅成功时）：
  进入 DIFF 比对模式
  ├─ 左栏：只读原稿
  └─ 右栏：AI 建议稿（可直接修改）

操作 DIFF 结果：
  点击「接受并合并」         POST /documents/{doc_id}/apply-polish
  ├─ 后端先备份原稿至快照    → 审计节点 SNAPSHOT(11)
  ├─ 覆写正文                → 审计节点 POLISH_APPLIED(21)
  └─ 清空建议稿字段
  前端：viewMode → SINGLE，清空 aiPolishedContent

  点击「丢弃」               POST /documents/{doc_id}/discard-polish
  前端：viewMode → SINGLE，原稿不变
```

### 1.3 国标排版阶段（可选手动触发）

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
点击「GB 国标排版并下载」    POST /api/v1/tasks/format
                            ← 返回 task_id
                            Worker 按 doc_type_id 加载文种排版规则（GENERAL 及统计类文种部分免检）
                            生成符合 GB/T 9704-2023 的 .docx 文件

任务完成后：
  通知卡片：「排版已完成，点击下载」
  按钮变为「下载国标文档 📥」

若结构不匹配文种要素：
  通知附加黄色警告列表       允许下载，但提示结构问题

若任务失败：
  公文详情页展示「排版失败」横幅
  显示「重新排版」按钮（橙色）
```

### 1.4 提交审批阶段 (SUBMITTED)

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
点击「提交审批」             POST /documents/{doc_id}/submit
                            （系统将保存正文、释放锁、更新状态统一为一次原子请求）
                            前置校验：status == DRAFTING（否则 409）
                            通过后：
                            ├─ status → SUBMITTED(30)
                            ├─ submitted_at 写入 approval_logs 记录
                            ├─ 审计节点 SUBMITTED(30)
                            └─ ← 返回 {doc_id, status: "SUBMITTED", log_id}

科室负责人侧：
  接到通知（铃铛图标出现未读角标）
  在 /approvals 页面看到待审批公文
```

### 1.5 签批审核阶段

```
科室负责人操作              系统响应
──────────────────────────────────────────────────────────────
打开待签批公文               POST /approval/{doc_id}/review
前置校验：status == SUBMITTED（否则 409）
权限校验：dept_head_id==current_user.user_id 或 (role_level>=5 且 dept_id==doc.dept_id)，且 is_active==True

审阅公文终版内容

选择「批准并签署」：
  弹出重度警戒确认弹窗       二次确认
  确认后：
  ├─ status → APPROVED(40)（终态，不可逆）
  ├─ NFKC 归一化流水线 → SIP 哈希存证
  ├─ reviewer_id 写入 documents 表
  ├─ document_approval_logs 记录完整
  ├─ After-Commit：自动派发 FORMAT 任务（携带 doc_type_id）
  └─ 审计节点 APPROVED(40)
  起草人侧：接收批准通知

选择「驳回打回」：
  弹出驳回理由必填对话框
  填写并确认后：
  ├─ status → REJECTED(41)
  ├─ rejection_reason 写入 approval_logs
  ├─ reviewer_id 写入 documents 表
  └─ 审计节点 REJECTED(41)
  起草人侧：接收驳回通知（含驳回理由）、「前往修改」按钮
```

### 1.6 驳回修改阶段 (REJECTED → DRAFTING)

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
点击「前往修改」             POST /documents/{doc_id}/revise
步骤 0：前置校验             status == REJECTED（否则 409）
步骤 1：原子性抢锁            SET lock:{doc_id} NX EX 180
         抢锁失败 → 409 "已有他人占锁"
         抢锁成功 → 继续
步骤 2：数据库事务内          status → DRAFTING
                             清空 ai_polished_content、draft_suggestion
步骤 3：审计                  节点 REVISION(42)
步骤 4：返回带锁凭证          lock_token、lock_expires_at

前端：带锁直接进入工作区编辑状态
      可重新 AI 润色、修改内容
      再次提交审批，进入 1.4 阶段
```

## 二、知识库资产管理工作流

### 2.1 文件上传流程

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
进入 /knowledge 页面
选择目标分仓：
  个人沙箱库 / 科室共享库 / 全局基础库

单文件/多文件/文件夹 拖拽：
  弹出"上传配置"抽屉
  必须选择「安全等级」：      一般 / 重要 / 核心
  点击「确认上传」

上传至目标目录：
  POST /api/v1/kb/upload      sha-256 哈希碰撞检测
  ├─ 命中且等级匹配 → 直接复用切片，节点 → READY
  ├─ 命中但等级不符（新等级更高）→ 强制重新解析
  └─ 未命中 → 创建新物理文件，节点 → UPLOADED
             → Celery PARSE 任务入队
             → 前端 SSE 实时监听：UPLOADED → PARSING → READY

PARSE 任务失败：
  节点标记为 FAILED（红色标签）
  显示「重新解析」按钮
```

### 2.2 替换上传流程

```
悬浮文件节点 → 「替换上传」
  新文件哈希 == 旧文件哈希 → 拦截，提示"内容无变化"
  新文件哈希 != 旧文件哈希 →
  ├─ SELECT ... FOR UPDATE 锁定旧节点
  ├─ UPDATE 旧节点（软删除旧切片）
  └─ INSERT 新节点 → 派发 PARSE 任务
```

### 2.3 软删除流程

```
悬浮文件节点 → 「删除」 → 二次确认弹窗
  确认后：
  DELETE /api/v1/kb/{kb_id}
  ├─ WITH RECURSIVE CTE 级联软删除目录及子节点
  ├─ 对应切片 embedding 置 NULL（从向量检索移除）
  └─ 审计日志记录影响的切片 ID
```

## 三、参考范文库管理工作流

### 3.1 上传范文（管理员/科室负责人）

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
/knowledge → 「参考范文」Tab
点击「上传新范文」
弹出上传 Drawer：
  选择 .docx 文件            仅接受 .docx 格式
  填写范文标题
  选择关联文种（必选）
  确认上传

POST /api/v1/exemplars/upload
├─ 校验 MIME 类型（application/vnd.openxmlformats...）
├─ SHA-256 去重检测
├─ 自动提取 extracted_text（供 AI few-shot）
│   提取失败 → 允许降级存入（extracted_text 置空），前端抛出警告但不阻断
└─ 入库，返回新 exemplar_id

权限说明：
  BASE 层：仅 role_level >= 99（管理员）可写
  DEPT 层：仅 dept_head_id（科室负责人）可写
```

### 3.2 起草时选择参考范文

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
在工作区左侧「参考范文」面板
  按当前公文文种自动过滤      GET /api/v1/exemplars?doc_type_id={id}
  Radio 单选一份范文          exemplar_id 写入 useEditorStore

点击「AI 智能润色」时：
  exemplar_id 随请求体提交    后端将范文 extracted_text 注入 few-shot
```

### 3.3 软删除范文

```
悬浮范文卡片 → 「删除」 → 二次确认
DELETE /api/v1/exemplars/{exemplar_id}
├─ 前置检查：是否有 DRAFTING 状态公文引用此范文
│   有引用 → 拒绝删除，提示"被 N 份草稿引用"
│   无引用 → 软删除（is_deleted = True）
└─ 已提交/审批的公文引用允许孤悬（不影响业务）
```

## 四、RAG 智能问答工作流

```
用户操作                    系统响应
──────────────────────────────────────────────────────────────
进入 /chat 页面
左侧勾选知识库节点（限定查询范围）
输入查询问题，选择「流式响应」
POST /api/v1/chat/stream
├─ 递归展开勾选节点下所有叶子切片
├─ 安全等级过滤（CORE 资产不参与）
├─ HNSW 向量检索 + BM25 混合召回
├─ 强制过滤 is_deleted=FALSE 切片
└─ SSE 流式推送回答
   ToolTips 展示引用的 CHUNK 坐标

未找到相关内容时：
  降级提示"未探明对应统计线索"
```

## 五、系统管理工作流（管理员，`role_level >= 99`）

本节涵盖所有在中枢设置（`/settings`）下执行的操作，部分查询功能对科长（`role_level >= 5`）开放。

### 5.1 用户与科室管理
- **科室CRUD**：创建、停用科室，指定 `dept_head_id`（科长）。删除前需检查关联。
- **用户CRUD**：创建账户、分配 `role_level`、绑定科室。
- **停用与密码重置**：停用用户（`is_active=False`）后，该用户所有请求返回 HTTP 403；管理员可为用户重置密码（需二次确认）。

### 5.2 公文文种与提示词管理
- **文种管理**：CRUD 维护 `document_types` 表，在线编辑 `layout_rules` JSON 规则。
- **提示词管理**：在线编辑 `app/prompts/` 目录下的系统提示词与词表，保存前二次确认，保存后自动触发热加载生效，操作计入审计。

### 5.3 审计与安全存证查询 (`role_level >= 5`)
- **审计日志**：多条件筛选查询 `nbs_workflow_audit` 记录。科长仅可查本科室，管理员查全局。
- **存证校验**：查询审批日志，通过 `GET /documents/{doc_id}/verify-sip` 即时校验文档哈希是否被篡改。

### 5.4 锁与任务大盘监控
- **锁监控**：查看 Redis 活跃锁，执行"强制释放"，向被驱逐者推送 `LOCK_RECLAIMED` SSE 事件使其进入只读模式。
- **任务监控**：监控全系统异步任务，查看 FAILED 任务堆栈，执行手动重试（`retry_count < 3`）或标记肃清。

### 5.5 系统健康与参数调优
- **系统探针**：查看数据库、Redis、Celery 节点、AI 引擎的连通性及服务器负载。
- **参数配置**：动态更新锁 TTL、心跳间隔、自动保存频率、限流阈值等参数，持久化至数据库。
- **系统清理**：触发废弃临时文件清理、孤立文件扫描或 GIN 索引死条目置空。

### 5.6 数据库快照管理
- **手工备份**：点击"立即备份"触发 `pg_dump` 异步任务。
- **快照恢复**：高危操作，需双重确认（输入 `CONFIRM`）后执行 `pg_restore`，全程录入 `CRITICAL` 审计日志。

## 六、异常与容灾恢复流程

### 6.1 网络断线/浏览器崩溃恢复
```
beforeunload 事件触发时：
  fetch(keepalive: true) → POST /locks/release
  （payload 同时携带最新正文内容与锁凭证，后端先保存再释放锁，合并为单一原子请求）

重新打开工作区时：
  ├─ 检查云端快照列表（document_snapshots）
  ├─ 若有 AI 生成的未决稿 → 强制进入 DIFF 比对模式
  └─ 从快照恢复 → 审计节点 SNAPSHOT_RESTORE(12)
```

### 6.2 排版失败恢复
```
公文详情页：「排版失败」横幅 + 「重新排版」按钮
点击「重新排版」            POST /api/v1/tasks/format
                            权限：属主（起草人）/ 管理员
                            重新派发 FORMAT 任务
```

### 6.3 登录踢出处理
```
同一账号在其他设备登录时：
  旧设备下次请求 → 401 Unauthorized
  前端全局拦截器 → 跳转 /login
  提示"您的账号已在其他设备登录"
```
