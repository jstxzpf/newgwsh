## 二、 基础设施编排约束 (Infrastructure & Deployment)

> 以下约定是《口语化约定》第一条"正式生产部署到 10.132.60.111 Ubuntu 24.04 Server"的具体落地规范，用于指导 Dockerfile 编写与 docker compose 编排。

### 1. Dockerfile 多阶段构建规范（最小化原则）

* **后端（FastAPI）**：必须采用多阶段构建。`builder` 阶段基于 `python:3.12-slim` 安装依赖；`runtime` 阶段仅复制已安装的 `site-packages` 与应用代码，**不包含** pip、构建工具链等。
* **前端（React/Vite）**：`builder` 阶段基于 `node:20-slim` 执行 `npm run build`；`runtime` 阶段使用 `nginx:alpine` 仅复制 `dist/` 静态产物。
* **基础镜像优先使用本机已有镜像**：Dockerfile 的 `FROM` 指令优先选用本机 `docker images` 中已存在的镜像 Tag，避免重复拉取基础镜像。
* **`.dockerignore` 必须配置**：排除 `node_modules/`、`.venv/`、`__pycache__/`、`*.pyc`、`.git/`、`app/data/outputs/`（运行时产物）、`app/prompts/`（由 Volume 挂载）等，防止上下文膨胀。

### 2. `docker compose` 网络隔离规范

生产环境 `docker-compose.yml` 必须定义两个独立网络，实现前后端与数据层的严格隔离：

```yaml
networks:
  frontend_net:   # 前端 Nginx 与后端 FastAPI 通信
    driver: bridge
  backend_net:    # 后端 FastAPI、Celery、PostgreSQL、Redis 内部通信
    driver: bridge
```

* `nginx`（前端）：仅连接 `frontend_net`，**不得直连** `backend_net`（不可访问 PG/Redis）。
* `fastapi`（后端网关）：同时连接 `frontend_net` 和 `backend_net`，作为前后端的唯一通信桥梁。
* `celery`（Worker）：仅连接 `backend_net`。
* `postgres` 和 `redis`：仅连接 `backend_net`，**严禁暴露端口到宿主机**（生产环境去掉 `ports` 映射）。
* `ollama`（AI 算力节点）：不纳入 compose 管理（本机独立部署），后端通过 `OLLAMA_BASE_URL` 环境变量访问，**路由守卫确保仅允许 `10.132.x.x` 内网访问**。

### 3. Volume 挂载粒度规范

```yaml
volumes:
  pg_data:            # PostgreSQL 数据持久化（命名 Volume，不使用 bind mount）
  redis_data:         # Redis AOF 持久化
  archive_root:       # pg_dump 备份归档（对应后端 ARCHIVE_ROOT）

services:
  fastapi:
    extra_hosts:
      - "host.docker.internal:host-gateway"  # 允许容器访问宿主机的 Ollama 节点
    volumes:
      - ./app/prompts:/app/prompts   # 提示词热加载（bind mount，支持管理员在线编辑后立即生效）
      - ./app/data:/app/data         # 知识库文件与排版产物（bind mount，便于运维直接访问）
      - archive_root:/app/archive    # 归档目录（命名 Volume 统一管理）
  postgres:
    volumes:
      - pg_data:/var/lib/postgresql/data
```

**关键约束**：
* `app/prompts/` 必须使用 **bind mount**（宿主机目录直接挂载），确保管理员通过 `POST /sys/reload-prompts` 热加载后，容器内的文件立即反映修改，无需重启容器。
* 数据库 Volume 必须使用 **命名 Volume**（不能是 bind mount），防止路径权限问题导致 PG 无法写入。

### 4. 国内镜像源配置规范

* **Python pip**：Dockerfile 中执行 `pip install` 时，统一附加 `-i https://pypi.tuna.tsinghua.edu.cn/simple --trusted-host pypi.tuna.tsinghua.edu.cn`。
* **npm**：前端 builder 阶段执行 `npm install` 前，先运行 `npm config set registry https://registry.npmmirror.com`。
* **Docker Hub 镜像拉取**：开发机 Docker Desktop 配置 `"registry-mirrors": ["https://mirror.ccs.tencentyun.com", "https://hub-mirror.c.163.com"]`（在 Docker Desktop 的 Docker Engine JSON 中配置）。
* **Ubuntu apt**：生产服务器 `/etc/apt/sources.list` 替换为阿里云源或华为云源（服务器首次初始化时操作，不在 Dockerfile 内执行，避免构建时访问外网）。

### 5. 开发环境与生产环境区分

* **开发（本机 Docker Desktop + Windows 11）**：使用 `docker-compose.dev.yml`，启用所有服务的端口映射（PG:5432、Redis:6379、FastAPI:8000），`volumes` 使用 bind mount 便于热重载（`uvicorn --reload`）。
* **生产（10.132.60.111 Ubuntu 24.04）**：使用 `docker-compose.yml`（生产配置），去掉 PG/Redis 端口对外暴露，FastAPI 运行 `--workers 4`（根据服务器 CPU 核心数调整），Nginx 监听 `80/443`。
* **部署命令**：生产环境统一使用 `docker compose up -d`（注意：`docker compose` 而非 `docker-compose`，服务器已配置 Compose V2 插件）。`&&` 命令连接符在 Windows 开发机 PowerShell 中不受支持，开发机脚本必须分行执行或使用 `;` 分隔。

### 6. 常见异常排查与修复 (Troubleshooting)

* **登录提示"系统异常" (500 Error)**：
    * **原因**：在 Docker Compose 环境下，`.env` 文件中的 `POSTGRES_HOST`、`REDIS_URL` 等若配置为 `localhost`，容器会尝试连接自身而非宿主机或其他容器，导致数据库/缓存连接失败。
    * **修复**：在 `.env` 中，将 `POSTGRES_HOST` 改为 `db`，将 `REDIS_URL` 中的 `localhost` 改为 `redis`。若需连接宿主机的 Ollama，将 `OLLAMA_BASE_URL` 改为 `http://host.docker.internal:11434`。
    * **验证**：修改 `.env` 后，执行 `docker compose up -d --force-recreate api` 重新加载配置。

* **AI 智能润色按钮死锁（遮罩无限加载）**：
    * **原因**：前端 `handlePolish` 在派发 Celery 异步任务后，仅将 `task_id` 存入 store，但缺失了针对任务状态（`getStatus`）的轮询或 SSE 监听。导致 `isBusy` 永远卡在 `true`。
    * **修复**：在 `Workspace.tsx` 的 `handlePolish` 中引入基于 `setInterval` 的状态轮询机制（每 3 秒检测一次），在任务返回 `COMPLETED` 或 `FAILED` 时及时调用 `setBusy(false)` 并刷新文档状态。

* **提交审批按钮被连带阻塞（操作互斥）**：
    * **原因**：提交按钮直接使用了 `loading={isBusy}` 属性，被上述未清除的 `isBusy` 连带卡死；且原本的代码缺乏互斥检查，允许在执行其他后台任务时点击提交。
    * **修复**：在 `handleSubmit` 中增加前置逻辑：`if (isBusy) { return message.warning('...'); }`，显式实现业务上的互斥保护。

* **E2E 物理测试运行异常（如 401 报错或等待元素超时）**：
    * **原因 1（401 报错）**：容器化重启后数据库被清空，或 `seed_data.py` / `reset_admin_pwd.py` 设置的管理员密码与 Playwright 测试脚本硬编码的密码不一致（例如 `admin123` 与 `Admin123!` 的区别）。
    * **修复 1**：跑 E2E 测试前需先登入 `api` 容器执行 `python scripts/seed_data.py` 和 `python scripts/prepare_physical_test.py`；并确保测试脚本的密码与重置密码保持一致。
    * **原因 2（断言超时）**：检查 Ant Design 按钮的 Loading 状态时，`locator('.ant-btn-loading-icon').toBeVisible()` 会因内部 CSS 动画（瞬态隐藏）导致测试 Flaky 失败。
    * **修复 2**：改用 `toHaveClass(/ant-btn-loading/)` 匹配按钮的 class 属性，绕过具体的子元素动画，提升自动化测试的稳定性。

* **浏览器自动化登录失败 (401 Unauthorized)**：
    * **原因**：脚本（如 `test_login.js`）或旧文档中硬编码的密码（如 `Admin123!`）与数据库实际初始化的密码（如 `admin123`）不一致。
    * **修复**：统一使用项目约定的初始密码。若不确定，可进入 `api` 容器运行 `python scripts/check_users.py`（需设置 `PYTHONPATH=.`）核实。
    * **验证**：Playwright 脚本运行后应确认跳转至 `/dashboard`。

* **本地脚本连接数据库失败 (DNS/Connection Error)**：
    * **原因**：当 `.env` 配置为 `POSTGRES_HOST=db`（容器名）时，在宿主机直接运行 Python 脚本会报错 `socket.gaierror: [Errno 11001] getaddrinfo failed`，因为宿主机无法解析 `db` 域名。
    * **修复**：在宿主机调试时，临时修改 `.env` 将 `POSTGRES_HOST` 改为 `localhost`（需确保已映射 5432 端口）；或使用 `docker exec -it <api_container> python scripts/xxx.py` 在容器内执行。
