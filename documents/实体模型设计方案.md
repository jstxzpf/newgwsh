# 国家统计局泰兴调查队公文处理系统 V3.0 - 实体模型设计方案

## 一、 核心存储与设计原则
1. **基础设施**：PostgreSQL 15 + `pgvector` 插件。
2. **ORM 映射**：使用 SQLAlchemy 2.0，表名统一采用下划线命名法（snake_case），并增加复数后缀。
3. **软删除策略**：对于公文、知识库等核心资产，均采用软删除（`is_deleted`）以满足国家统计局的数据保留审计合规要求。
4. **特殊数据类型**：
   * 树形层级数据（知识库目录）：采用 `parent_id` 邻接表模型。
   * 非结构化特征（表格坐标、页面来源）：采用 `JSONB` 格式实现 Schema-Free 灵活存储。
   * AI 特征数据：采用 `VECTOR` 类型存储文本 Embeddings，并建立 `HNSW` 向量索引。
5. **不可篡改日志**：审计表（`nbs_workflow_audit`）与签批存证表（`document_approval_logs`）仅允许 `INSERT`（Append-Only），严禁 `UPDATE` 或 `DELETE`。

## 二、 全局变量字典（枚举与常量）

本系统的核心业务状态由严格的枚举（Enum）约束，在代码层和数据库层保持强一致。

### 1. DocumentStatus (公文生命周期状态)
| 枚举值 | 说明 | 允许的后续状态 |
| :--- | :--- | :--- |
| `DRAFTING` | 起草编辑中 | `SUBMITTED` |
| `SUBMITTED` | 已提交待审批 | `APPROVED`, `REJECTED` |
| `APPROVED` | 已签批通过（终态） | — |
| `REJECTED` | 已驳回待修改 | `DRAFTING`（重新编辑后回到起草态） |

### 2. KBTier (知识库层级)
| 枚举值 | 说明 | 权限控制规则 |
| :--- | :--- | :--- |
| `BASE` | 基础全局库 | 仅 `role_level >= 99`（系统管理员）可写入，全员只读。存放《统计法》、公文国标等。 |
| `DEPT` | 科室共享库 | 仅 `role_level >= 5`（科室负责人）可写入，本科室成员只读。 |
| `PERSONAL`| 个人沙箱库 | 仅属主（Owner）可读写，不可跨用户越权。 |

### 3. DataSecurityLevel (数据安全等级)
| 枚举值 | 说明 | 防护策略 |
| :--- | :--- | :--- |
| `CORE` | 核心涉密数据 | 禁止直接作为 Prompt 上下文传输给外部或边缘算力节点，仅允许本地校验。后端 RRF 融合后，若废弃了 CORE 切片，**必须在安全审计日志中记录"因安全等级限制，本次检索废弃 N 条 CORE 切片"**，但返回给用户的提示必须与"未找到数据"（即"未探明对应线索"）完全一致，**严禁透露因安全等级过滤而导致的搜索结果差异**，防止用户通过"能否召回"反推数据密级（侧信道信息泄露）。**前端知识库界面**：`CORE` 等级的文件应显示明显的"🔒 涉密，不可用于 AI 问答"标记，令用户事先知晓该资产不参与 RAG 检索，避免产生困惑。 |
| `IMPORTANT`| 重要统计数据 | 允许通过 RAG 挂载，但前端强制开启高频防泄密水印。 |
| `GENERAL` | 一般公开数据 | 正常流转与引用。 |

### 4. WorkflowNode (工作流审计节点)

> 编号规则：`1x` = 起草阶段，`2x` = AI 辅助阶段，`3x` = 审批阶段，`4x` = 终态阶段。

| 枚举值 | 节点代号 | 说明 | 触发时机 |
| :--- | :--- | :--- | :--- |
| `10` | `DRAFTING` | 拟稿起草 | 后端 `POST /api/v1/documents/init` 创建公文并分配 `doc_id` 时唯一触发。 |
| `11` | `SNAPSHOT` | 云端快照备份 | 执行"接受润色"、快照恢复等高危覆写操作前自动触发备份时记录。 |
| `12` | `SNAPSHOT_RESTORE` | 快照恢复 | 用户从历史快照中选择恢复时触发。 |
| `20` | `POLISH_REQUESTED` | AI 润色任务派发 | 用户点击"AI 智能润色"，后端成功派发 Celery 润色任务时触发。 |
| `21` | `POLISH_APPLIED` | AI 润色建议接受 | 用户点击"接受并合并"，正文被润色内容覆写后触发。 |
| `22` | `FORMAT_REQUESTED` | 排版任务派发 | 审批通过后自动派发或管理员手动触发排版任务时记录。 |
| `23` | `FORMAT_COMPLETED` | 排版产物生成 | Celery 排版 Worker 成功生成 `.docx` 文件时触发。 |
| `30` | `SUBMITTED` | 提交审批 | 起草人将公文提交至科室负责人审批时触发。 |
| `40` | `APPROVED` | 签批通过 | 科室负责人审阅通过并执行签批动作时触发（终态）。 |
| `41` | `REJECTED` | 签批驳回 | 科室负责人驳回公文并填写驳回理由时触发。驳回理由存入 `action_details`。 |
| `42` | `REVISION` | 驳回修改 | 起草人在公文被驳回后重新打开编辑时触发。公文回到 `DRAFTING` 状态，系统原子性完成锁抢占。 |

### 5. DocType (公文文种)
> GB/T 9704-2023 规定的主要行政公文文种及统计调查专用文种，决定排版要素与结构规则。

| 枚举值 | 说明 | 核心排版要素 |
| :--- | :--- | :--- |
| `NOTICE` | 通知 | 通知缘由、通知事项、执行要求、落款 |
| `REQUEST` | 请示 | 请示缘由、请示事项、结尾请批语（"妥否，请批示"） |
| `REPORT` | 报告 | 报告缘由、报告事项、结尾报告语（"特此报告"） |
| `REPLY` | 批复 | 批复引据、批复意见、执行要求 |
| `LETTER` | 函 | 函告事由、函告事项、结尾请复语 |
| `MINUTES` | 纪要 | 会议基本情况、会议议定事项、参会人员 |
| `RESEARCH` | 调研分析 | 调研背景、调研方法、主要发现、分析结论、政策建议 |
| `ECONOMIC_INFO` | 经济信息 | 信息摘要、核心数据指标、趋势研判、附注数据来源 |
| `GENERAL` | 通用文档 | 无强制要素要求，仅执行基础排版（字体/行距/页边距/首行缩进）。适用于工作总结、方案、简报等非正式行政文体 |

### 6. TaskType (异步任务类型)
| 枚举值 | 说明 |
| :--- | :--- |
| `POLISH` | AI 语义润色任务 |
| `FORMAT` | GB/T 标准排版任务 |
| `PARSE` | 知识库文件解析入库任务 |

### 7. TaskStatus (异步任务状态)
| 枚举值 | 说明 |
| :--- | :--- |
| `QUEUED` | 已入队等待执行 |
| `PROCESSING` | 正在执行中 |
| `COMPLETED` | 执行成功 |
| `FAILED` | 执行失败 |

## 三、 数据库结构详细设计 (Table Schema)

### 1. 科室组织表 (`departments`)
**描述**：泰兴市调查队的科室组织架构，支持按科室维度进行权限控制、数据统计与审计追踪。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `dept_id` | Integer | Primary Key | Auto Inc | 科室唯一主键 |
| `dept_name` | String(128) | Unique, Not Null | None | 科室名称（如：执法监督室） |
| `dept_code` | String(32) | Unique | None | 科室编码（如：ZF01），用于程序化引用 |
| `dept_head_id`| Integer | FK(`system_users.user_id`), Nullable | `Null` | 预留扩充：科室最高负责人绑定 ID，供刚性审批流备用 |
| `is_active` | Boolean | Not Null | `True` | 科室启停状态 |
| `created_at` | DateTime | Not Null | `now()` | 创建时间 |

### 2. 系统用户表 (`system_users`)
**描述**：存储泰兴市调查队全体干部的基本信息与 RBAC 权限基线。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `user_id` | Integer | Primary Key | Auto Inc | 用户唯一主键 |
| `username` | String(64) | Unique, Index | None | 统一登录工号 |
| `full_name` | String(64) | Not Null | None | 用户真实姓名（用于水印、锁冲突提示"XX 正在编辑"、审批展示等） |
| `password_hash`| String(255)| Not Null | None | 密码哈希值（**必须使用 Argon2id 算法**，严禁使用 SHA-256/MD5 等快速哈希） |
| `dept_id` | Integer | FK(`departments.dept_id`), Index | None | 所属科室 ID |
| `role_level` | Integer | Not Null | `1` | 权限阈值：`99`管理员，`5`科长，`1`科员 |
| `is_active` | Boolean | Not Null | `True` | 账号启停状态 |
| `created_at` | DateTime | Not Null | `now()` | 账号创建时间 |

### 2.5 用户会话表 (`users_sessions`)
**描述**：存储用户登录后的 Refresh Token 及会话元数据，支持单一设备登录控制与强制踢出。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `session_id` | String(64) | Primary Key | None | 会话 ID |
| `user_id` | Integer | FK(`system_users.user_id`), Index | Not Null | 所属用户 ID |
| `refresh_token_hash`| String(255)| Not Null | None | Refresh Token 的哈希值 |
| `device_info`| String(255)| Nullable | `Null` | 登录设备与浏览器指纹信息 |
| `expires_at` | DateTime | Index | Not Null | 会话过期时间 |
| `created_at` | DateTime | Not Null | `now()` | 会话创建时间 |

> **会话清理机制**：当用户主动调用 `POST /logout` 时，系统必须物理删除该 `session_id` 对应的记录，并同步清理 Redis 中的 Access/Refresh Token 缓存。

### 3. 公文主表 (`documents`) —— **【核心业务主表】**
**描述**：存储公文的全生命周期核心数据，是整个系统业务流转的中枢实体。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `doc_id` | String(64) | Primary Key | None | 后端生成的 UUID 公文追踪号（**由后端 `POST /api/v1/documents/init` 统一分配**） |
| `title` | String(255)| Not Null | `'未命名公文'` | 公文标题 |
| `content` | Text | Nullable | `''` | 公文 Markdown 正文内容 |
| `status` | Enum | Index, Not Null | `DRAFTING` | 见变量字典 `DocumentStatus`。ORM 层须增加 `@validates('status')` 钩子强制合法转换路径校验，非法转换抛出 HTTP 409。 |
| `doc_type_id` | Integer | FK(`document_types.type_id`), Not Null | None | 公文文种（通知/请示/报告等）。创建时必选；状态变为 `SUBMITTED` 后锁定不可修改。 |
| `exemplar_id` | Integer | FK(`exemplar_documents.exemplar_id`), Nullable | `Null` | 起草时选择的参考范文 ID。AI 润色时作为 few-shot 模板注入 System Prompt，不参与 RAG。 |
| `dept_id` | Integer | FK(`departments.dept_id`), Index | None | 发文科室 |
| `creator_id` | Integer | FK(`user_id`), Index | Not Null | 起草人 ID |
| `ai_polished_content`| Text | Nullable | `Null` | AI 润色后的建议稿（用于 DIFF 对比），接受或丢弃后清空 |
| `draft_suggestion` | Text | Nullable | `Null` | **极端灾备恢复字段**：在 DIFF 模式下自动保存时，存储用户对建议稿的二改草稿，保护工作进度 |
| `word_output_path`| String(512)| Nullable | `Null` | 最终排版 Word 文件的服务器路径 |
| `reviewer_id` | Integer | FK(`user_id`), Nullable | `Null` | **性能冗余字段**：记录最终审批通过或驳回的负责人 ID，加速科室公文列表检索 |
| `is_deleted` | Boolean | Not Null | `False` | 软删除标志 |
| `created_at` | DateTime | Not Null | `now()` | 创建时间 |
| `updated_at` | DateTime | Not Null | `now()` | 最后修改时间（后端自动保存时更新） |

### 4. 知识库目录树 (`knowledge_base_hierarchy`)
**描述**：统一管理多模态统计资产（支持多级文件夹嵌套），是 RAG 引擎的物理防波堤。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `kb_id` | Integer | Primary Key | Auto Inc | 知识资产 ID |
| `parent_id` | Integer | FK(`kb_id`), Index| `Null` | 父节点 ID，空表示根节点 |
| `kb_name` | String(255)| Not Null | None | 文件名或目录名称 |
| `kb_type` | Enum | Not Null | None | 资产类型：`FILE` 或 `DIRECTORY` |
| `kb_tier` | Enum | Not Null | `PERSONAL`| 见变量字典 `KBTier` |
| `dept_id` | Integer | FK(`departments.dept_id`), Index | Nullable | **归属科室**：上传时从用户当前科室快照写入，作为归属科室的不可变记录。 |
| `security_level`| Enum | Not Null | `GENERAL` | 见变量字典 `DataSecurityLevel` |
| `parse_status` | Enum | Not Null | `UPLOADED` | 解析状态：`UPLOADED` (已传), `PARSING` (解析中), `READY` (就绪), `FAILED` (失败) |
| `physical_file_id`| Integer | FK(`knowledge_physical_files.file_id`), Index | `Null` | **关联引用**：指向底层的物理文件记录。 |
| `owner_id` | Integer | FK(`user_id`),Idx| None | 上传者/属主 |
| `version` | Integer | Not Null | `1` | 资产版本号（替换上传时递增） |
| `is_deleted` | Boolean | Not Null | `False` | 软删除标志 |
| `deleted_at` | DateTime | Nullable | `Null` | 软删除时间戳 |
| `created_at` | DateTime | Not Null | `now()` | 资产入库时间 |
| `updated_at` | DateTime | Not Null | `now()` | 最后变更时间（用于 context_snapshot_version 校验） |

> **级联软删除策略**：当对一个 `DIRECTORY` 类型节点执行软删除时，在后端的 SQLAlchemy DAO 层，必须硬性规范使用 `WITH RECURSIVE` CTE（公用表表达式）。先在数据库引擎内部递归计算出所有需要波及的子代 `kb_id` 集合，再将其作为一个主键列表包裹在同一个数据库事务中，对目录表和切片表统一执行 `UPDATE`，**这涵盖将两者的 `is_deleted` 置为 `True`，并同时把切片表中 `embedding` 字段设为 `NULL`**（从向量索引中完全移除防幽灵查出）。仅保留 `content` 文本内容以满足审计留痕要求。

### 5. 知识特征向量表 (`knowledge_chunks`) —— **【核心重构表】**
**描述**：将知识库文件切片（Chunking）后生成的细粒度数据块及高维向量存储，直接支撑 RAG 召回。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `chunk_id` | Integer | Primary Key | Auto Inc | 切片唯一标识 |
| `kb_id` | Integer | FK(`kb_id`), Index| Not Null | 关联的原始知识资产逻辑 ID |
| `physical_file_id`| Integer | FK(`knowledge_physical_files.file_id`), Index | Not Null | **冗余关联**：指向底层的物理文件记录。支持当不同用户或同用户上传相同物理文件时直接进行切片复用，避免重复入库。 |
| `chunk_index` | Integer | Not Null | None | 该切片在原文档中的顺序索引 |
| `content` | Text | Not Null | None | 文本切片内容全文（用于 BM25 检索） |
| `embedding` | Vector(1024) | HNSW Index | None | 适配 bge-m3 中文增强模型，标准嵌入维度 1024 维。用于计算余弦相似度。 |
| `is_deleted` | Boolean | Index, Not Null| `False` | 冗余字段：同步母文档软删除状态，RAG 召回强制过滤 |
| `kb_tier` | Enum | Not Null | None | 冗余字段：知识库层级（支持快速越权拦截） |
| `security_level`| Enum | Not Null | None | 冗余字段：数据安全等级（支持快速涉密熔断） |
| `dept_id` | Integer | FK(`departments.dept_id`), Index | Nullable | **冗余字段**：从 `knowledge_base_hierarchy.dept_id` 冗余复制，支持跨用户科室变动后的权限锁定。 |
| `metadata_json`| JSONB | GIN Index | `{}` | **结构化挂载库**：存储表格行列坐标、页码等。其中必须强制注入切片的**完整层级标题路径**（如 `{"title_path": "统计法|第三章|第十条"}`）。若为 Tabular RAG 产物，需包含行摘要文本及 `{"row_data": {...}}`。 |

### 6. 工作流合规审计表 (`nbs_workflow_audit`)
**描述**：公文生命周期流水账（Event Sourcing）。记录不可修改，仅允许追加（Append-Only），用于后期定责。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `audit_id` | Integer | Primary Key | Auto Inc | 流水号 |
| `doc_id` | String(64) | FK(`documents.doc_id`), Index | Not Null | 关联的公文追踪号 |
| `workflow_node_id`| Integer | Not Null | None | 见变量字典 `WorkflowNode` |
| `operator_id` | Integer | FK(`user_id`) | Not Null | 触发该节点的执行人 |
| `reference_id` | Integer | FK(`approval_logs.log_id`), Nullable | `Null` | **审批关联 ID**：指向审批日志记录，支持在多次循环中精确回溯审批意见 |
| `action_details`| JSONB | Nullable | `Null` | 补充信息（如 AI 润色的提示词模板编号、驳回理由等） |
| `action_timestamp`| DateTime| Index | `now()` | 动作发生的精确时间 |

### 7. 公文签批存证表 (`document_approval_logs`)
**描述**：闭环管理的终态存储池。一旦公文落入此表且状态为 `APPROVED`，该 `doc_id` 的所有生命周期即告终结。
| 字段名 | 数据类型 | 约束/索引 | 默认值 | 描述说明 |
| :--- | :--- | :--- | :--- | :--- |
| `log_id` | Integer | Primary Key | Auto Inc | 归档号 |

## 四、 实体索引优化建议 (Database Tuning)

在通过 SQLAlchemy 执行 DDL 语句创建上述模型时，强烈建议建立以下混合索引结构，以保障几十万条统计台账的毫秒级召回：

1. **向量检索加速 (pgvector)**：
   对 `knowledge_chunks.embedding` 字段建立 HNSW 索引。
   **双重防波堤设计**：为防止级联删除后的幽灵检索，DAO 层在 RAG 检索时必须强制执行 `JOIN knowledge_base_hierarchy h ON chunks.kb_id = h.kb_id` 并在最外层叠加 `WHERE h.is_deleted = FALSE`。
   ```sql
   CREATE INDEX ON knowledge_chunks USING hnsw (embedding vector_cosine_ops) WHERE is_deleted = FALSE;
   ```
2. **文本混合召回加速 (BM25)**：
   对 `knowledge_chunks.content` 添加 GIN 全文检索索引，以便配合向量检索实现精确的 HRAG 混合打分：
   ```sql
   CREATE INDEX ON knowledge_chunks USING gin(to_tsvector('zh', content));
   ```
3. **结构化穿透加速 (JSONB)**：
   对 `metadata_json` 施加 GIN 索引，加速对特定"表格列名"或"页面属性"的穿透定位：
   ```sql
   CREATE INDEX ON knowledge_chunks USING gin(metadata_json);
   ```
4. **公文状态复合索引**：
   对 `documents` 表建立科室+状态的复合索引，加速"查看本科室待审批公文"等高频查询：
   ```sql
   CREATE INDEX ON documents (dept_id, status) WHERE is_deleted = FALSE;
   ```
5. **软删除过滤索引与哈希引用约束**：
   对 `knowledge_base_hierarchy` 表建立排除已删除记录的查询优化部分索引，并增加针对个人库的防重复引用约束（基础库/科室库防重由业务接口逻辑保障，不作 DB 层约束）：
   ```sql
   CREATE INDEX ON knowledge_base_hierarchy (owner_id, kb_tier) WHERE is_deleted = FALSE;
   CREATE UNIQUE INDEX idx_kb_personal_unique ON knowledge_base_hierarchy(physical_file_id, owner_id) WHERE kb_type='FILE' AND kb_tier='PERSONAL' AND is_deleted=FALSE;
   ```
