# 泰兴市国家统计局公文处理系统 V3.0 - 特殊约定与开发铁律

## 一、 数据库与会话隔离铁律 (Database & Session Isolation)

### 1. 严格的异步与同步物理隔离
由于系统在同一个代码库中混合了高并发 HTTP 网关（FastAPI）与重型后台计算（Celery），数据库会话的调用方式是不可逾越的红线：
* **路由层（API 层）**：必须且只能使用 `AsyncSessionLocal` (基于 `asyncpg` 引擎) 处理前端请求。
* **后台任务（Celery `worker.py`）**：必须且只能使用 `SyncSessionLocal` (基于 `psycopg2` 引擎)。**严禁在 Worker 中使用异步会话**，否则会导致协程事件循环直接崩溃。

### 2. 强制软删除与不可篡改日志
* 针对 `knowledge_base_hierarchy` (知识库) 与核心公文表（`documents`），严禁在代码层执行 `session.delete()`。必须采用更新 `is_deleted = True` 的软删除策略，以符合国家统计局数据留存审计要求。
* 针对 `nbs_workflow_audit` (审计表) 和 `document_approval_logs` (审批存证表)，仅允许 `INSERT` 操作（Append-Only）。任何对其执行 `UPDATE` 或 `DELETE` 的 ORM 动作均属严重安全违规。

### 3. 知识库级联删除与双保险检索契约
前端发起删除指令时，后端执行级联软删除：
* **DAO 层统一入口**：所有 `is_deleted` 变更必须收束。
* **物理防线**：软删除时强制令 `knowledge_chunks.embedding = NULL`。
* **逻辑防线 (RAG)**：在执行向量检索或 BM25 查询时，**必须强制 `JOIN knowledge_base_hierarchy` 并过滤 `h.is_deleted = FALSE`**。仅依赖切片表单表过滤属于【低级安全隐患】。
（注：硬清空将 `embedding` 置 NULL 属于纯净逻辑删除与底线防御；而保留 HNSW 的 `WHERE is_deleted = FALSE` 的索引条件属于性能优化兼首道拦截。两者互为兜底共同构建双保险协同拒止体系，不视为冗余。同时，`knowledge_chunks` 表中的 `kb_tier`、`dept_id` 等冗余控制字段属于“不可变字段 (Immutable)”，一旦生成切片即固化，严禁通过 SQL 执行级联更新，如有权限变更需求必须走先软删除后重新解析切片入库的轮替流程）。

### 4. 结构化语义切片准则
* **语义锚点保护**：对于统计法规等结构化文档，切片算法必须识别标题层级（H1-H6）。每个切片块除正文外，必须在元数据中强制注入其所属的**完整标题路径锚点**。禁止出现将标题与直接从属条文物理切割在不同 Chunk 中的现象。
* **解析引擎一致性**：切片引擎必须与物理排版所用的 Markdown AST 解析逻辑保持高度一致，**全线代码明确锁定唯一授权使用 `markdown-it-py` 解析库生成抽象语法树**，严禁模块私自引入多重异构引擎，确保“所见即所分”。

### 5. 审计与事务隔离规约
* **审计非阻塞性**：`nbs_workflow_audit` 的写入严禁置于业务逻辑的主事务中以防事务污染。必须使用异步任务（Celery）或在 `After-Commit` 数据库钩子中触发。注意：该表中包含驳回理由等敏感明文，其可见性仅限起草人和涉事审批人，超管等非涉事人员查阅日志必须通过授权系统并审计其查阅行为。
* **版本替换并发锁与防重机制**：执行知识库文件替换时，必须先 `UPDATE` 逻辑树中的旧节点，再 `INSERT` 新节点，严禁利用 Unique Index 做 UPSERT 绕过。事务内必须引入 `SELECT ... FOR UPDATE` 锁定目标行。物理文件的防重复必须通过底层的 `knowledge_physical_files.content_hash` 全局唯一索引硬性保障；而逻辑记录的防重则在业务层约束（`BASE`/`DEPT` 检测同层级，`PERSONAL` 依赖部分唯一约束：`CREATE UNIQUE INDEX idx_kb_personal_unique ON knowledge_base_hierarchy(physical_file_id, owner_id) WHERE kb_type='FILE' AND kb_tier='PERSONAL' AND is_deleted=FALSE;`）。替换时若新上传文件算出的哈希等同于旧物理文件，则拦截阻断（内容无变化）。

### 6. 安全令牌与通讯指纹
* **令牌续期**：必须实现 Access/Refresh 双 Token 机制。Refresh Token 必须存放在 HttpOnly Cookie 中（必须同时配置 `Secure=True` 且 `SameSite=Strict`，严防跨站请求伪造 CSRF 攻击，仅允许在 HTTPS 承载下工作）。
* **SSE 权限锚定**：SSE Ticket 申请时必须在 Redis 中绑定 `task_id` 与 `user_id`，连接握手时必须校验该绑定关系。
* **签名一致性**：SIP 指纹计算严格采用公式 `normalized_content | reviewer_user_id | reviewed_at_iso8601`。归一化逻辑（NFKC + 去首尾空格 + 空格压缩）必须在各模块间严格复用。需注意当前机制防范了“终稿被偷偷修改”，但并不能防范“数据库运维直接修改终稿文本的同时，顺手重新计算了一次 SIP 并覆盖”，如需防范该场景则未来必须对接局内专用的外部 HSM 或时间戳防伪服务器。

## 二、 网络路由与算力安全约束 (Network & AI Compute Security)

### 1. 算力节点内网穿透阻断
系统核心驱动为部署在局域网的 Ollama LLM 引擎 (`10.132.60.133`)。
* **隔离策略**：严禁前端代码发起对大模型节点的直连。
* **后端守卫与限流**：所有大模型请求必须通过后端的 FastAPI 服务进行代理，且后端必须配置 `Routing Guard`，仅放行来源为 `10.132.x.x` 或 `127.0.0.1` 的内部流量，防止跨网段的非法算力盗用与数据刺探。同时，必须在后端网关层对 `POST /chat` 及 `POST /tasks/polish` 实施基于 Redis Token Bucket 的严格限流机制（如 5次/分钟/用户），拦截超出抛出 429。

### 2. 全局防泄密水印强制挂载
* 前端根视图必须强制挂载 `AntiLeakWatermark` 组件。该组件需动态提取当前用户的工号、所在科室名及当前时间（若后端有暴露 IPv4 则可增加网络层 IP），以 `0.08` 的极低透明度斜向覆盖全屏（`pointer-events: none`），确保所有截图行为均可被溯源定位。

## 三、 UI/UX 拟物与物理排版契约 (UI & Formatting Conventions)

### 1. 国标 A4 物理约束
前端公文编辑工作区（`EditorWorkspace.tsx`）**严禁**使用响应式宽度的 `width: 100%` 或流式布局。
* **尺寸锁定**：编辑器画板的基准物理宽度必须绝对锁定为 `794px`（A4 纸张的 96DPI 标准转换）。
* **动态防变形**：通过监听外部滚动容器宽度，动态计算并应用 `transform: scale()` 进行整体缩放。以此保障国标 GB/T 9704-2023 中规定的**"单行 28 个全角字符"**的硬性排版要求不被破坏、不出现意外换行。

### 2. 政务字体防御链
全局 CSS 变量中，针对公文正文区域的 `font-family` 设定必须严格按照以下降级顺序排列：
`font-family: "方正仿宋_GBK", "FZFS", "仿宋_GB2312", "仿宋", "FangSong", "STFangsong", "华文仿宋", "Noto Serif CJK SC", serif;`
必须将信创环境和政务内网的特定中文字体放在首位，严禁回退或使用系统默认的无衬线黑体（Sans-serif）渲染公文正文。

## 四、 鉴权隧道与防并发冲突契约 (Auth & Concurrency)

### 1. SSE 阅后即焚票据安全约束
原生 SSE 无法附加 `Authorization` Header，但**坚决禁止通过任何形态直接使用令牌做 URL 查询连接 (如：`?token=eyJ...`)，一经发现直接定性为中高级生产漏洞拦截。**
* **交换协定**：所有 SSE 长连接前置要求利用 Bearer Header 访问 `POST /api/v1/sse/ticket` 开具具备唯一且 TTL 少于 15 秒存活极限的换领票根 UUID `Ticket` 并发往前端。**开具票据前，后端必须提取 `Bearer Token` 验证身份，比对 Redis 中存储的 `task_owner:{task_id}` 是否与当前用户匹配，不匹配则拒绝颁发。**
* **信道贯通**：前置换发获票后，经 `?ticket={Ticket}` 叩通对应任务引擎建立管道。握手解析一旦验证确认立刻清除缓存。后端在处理 Ticket 校验时，**必须且只能执行一次性销毁动作（DEL key）**，确保 Ticket 在校验通过后即刻失效，彻底阻断同一 Ticket 的重播利用。

### 2. Redlock 唯一真理源与断网容灾释放
* **前置审批状态网关**：触发争抢与编辑前，锁路由必须阻隔非起草 (`DRAFTING`) 阶段文件的编辑态下探请求。
* **拒绝双规脑裂**：获取及控制该阶段并发必须以 Redis 的状态为 `Single Source of Truth`，数据库剥离此项管理职责。**在执行 `revise` 操作进行状态回退时，必须通过 Redis 的 `NX EX 180` 原子命令强制抢锁，彻底根除获取锁与状态变更间的竞态条件。**
* **防丢失强兜底**：前端退出或者卸载销毁前，不得单独发送释放锁通讯。强制将目前缓存最新的业务实体封装塞入 payload 向心跳通道抛掷并同时命令放弃持锁（**必须利用 `fetch` 搭配 `keepalive: true` 属性**执行，严禁使用无法携带自定义 Header 的 `sendBeacon`），以此防止自动间隔导致的边缘工作损失。

## 五、 全局防弹与容灾自愈 (Bulletproof & Recovery)

### 1. 前端报错防吞噬原则
在前端的请求拦截器（`apiClient`）或具体的业务 Try-Catch 块中，**严禁**出现仅使用 `console.log(error)` 的静默吞没行为。
* 除 `401 Unauthorized`（统一拦截并交由全局路由至 `/login`）外，所有 HTTP 4xx 和 5xx 的后端报错（尤其是后端的 `detail` 字段），必须通过 Ant Design 的 `message.error` 原样透传并弹窗展示给用户。

### 2. 全节点多轨漫游防灾
公文必须防患多端切换编辑的不安全隐患，拒绝以客户端本地 `Local Storage` 成为灾难备份的全部支撑点。**需特别明确边界**：本地缓存池内仅存留最近 2 条记录，这属于解决编辑器崩溃的极微秒级临时状态极速恢复机制，绝对不属于灾难备份策略，系统仍且必须以云端 `document_snapshots` 快照为主存储的唯一灾难恢复受信任数据源。
* **云灾备系统**：对系统执行"应用AI排版重写"等单边高重写风险动作前，并行派发一条写入远端数据图谱的 `POST` 请求（`document_snapshots`）。即使客户端粉碎换台仍可在侧边栏拉取复旧容灾操作。
* **漫游 DIFF 识别**：若异地调取文档发现其含有后生成未决的 `AI` 同步段落值，强制复原渲染成对比 DIFF 控制板，摆脱客户端 Local 的状态绑架。

## 六、 规范化运维与可观测性守护 (Monitoring & SOP)

系统的状态不能处于盲测之中。后端代码合入主干前必须满足监控检查清单：
* **结构化日志**：各层（如 `api`, `services`, `tasks` 等）关键出口必须部署基于 `structlog` 渲染的标准化 JSON 日志，包涵发生时刻、模块节点及 `doc_id`/`task_id`，并必须携带全局唯一的 `trace_id` 将孤立环节串联为全链路流。
* **高危操作探针**：涉及 `审批判决`、`文档清除` 及 `权限变动` 的接口必须显式吐出可采集的 `INFO` 或 `WARNING` 日志以便审计系统抓取。
* **算力熔断告警**：连接 Redis 及 Ollama 的服务句柄应当配置自动超时；Celery 死信队列及 `async_tasks` 中的滞留 FAILED 任务过载（如 5 分钟内错误超标）时，需抛出含 Traceback 的 `CRITICAL` 告警。
