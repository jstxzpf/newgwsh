# 泰兴市国家统计局公文处理系统 V3.0 - 特殊约定与开发铁律

## 一、 数据库与会话隔离铁律 (Database & Session Isolation)

### 1. 严格的异步与同步物理隔离
由于系统在同一个代码库中混合了高并发 HTTP 网关（FastAPI）与重型后台计算（Celery），数据库会话的调用方式是不可逾越的红线：
* **路由层（API 层）**：必须且只能使用 `AsyncSessionLocal` (基于 `asyncpg` 引擎) 处理前端请求。
* **后台任务（Celery `worker.py`）**：必须且只能使用 `SyncSessionLocal` (基于 `psycopg2` 引擎)。**严禁在 Worker 中使用异步会话**，否则会导致协程事件循环直接崩溃。

### 2. 强制软删除与不可篡改日志
* 针对 `knowledge_base_hierarchy` (知识库) 与核心公文表（`documents`），严禁在代码层执行 `session.delete()`。必须采用更新 `is_deleted = True` 的软删除策略，以符合国家统计局数据留存审计要求。
* 针对 `nbs_workflow_audit` (审计表) 和 `document_approval_logs` (审批存证表)，仅允许 `INSERT` 操作（Append-Only）。任何对其执行 `UPDATE` 或 `DELETE` 的 ORM 动作均属严重安全违规。

### 3. 知识库级联删除与双保险检索契约
前端发起删除指令时，后端执行级联软删除：
* **DAO 层统一入口**：所有 `is_deleted` 变更必须收束。
* **物理防线**：软删除时强制令 `knowledge_chunks.embedding = NULL`。
* **逻辑防线 (RAG)**：在执行向量检索或 BM25 查询时，**必须强制 `JOIN knowledge_base_hierarchy` 并过滤 `h.is_deleted = FALSE`**。仅依赖切片表单表过滤属于【低级安全隐患】。

### 4. 结构化语义切片与编辑器序列化
* **语义锚点保护**：对于统计法规等结构化文档，切片算法必须识别标题层级（H1-H6）。每个切片块除正文外，必须在元数据中强制注入其所属的**完整标题路径锚点**。禁止出现将标题与直接从属条文物理切割在不同 Chunk 中的现象。
* **解析引擎一致性**：切片引擎必须与物理排版所用的 Markdown AST 解析逻辑保持高度一致，**全线代码明确锁定唯一授权使用 `markdown-it-py` 解析库生成抽象语法树**，严禁模块私自引入多重异构引擎，确保“所见即所分”。
* **WYSIWYG 序列化底线**：无论前端采用何种可视化编辑器（如 Milkdown），后端存储字段 `documents.content` **必须始终为标准 Markdown 文本**。前端必须在提交/保存前将 DOM 状态序列化为纯净 Markdown，严禁将 HTML 标签持久化至数据库。

### 5. 审计与事务隔离规约
* **审计非阻塞性**：`nbs_workflow_audit` 的写入严禁置于业务逻辑的主事务中以防事务污染。必须使用异步任务（Celery）或在 `After-Commit` 数据库钩子中触发。注意：该表中包含驳回理由等敏感明文，其可见性仅限起草人和涉事审批人，超管等非涉事人员查阅日志必须通过授权系统并审计其查阅行为。
* **版本替换并发锁与防重机制**：执行知识库文件替换时，必须先 `UPDATE` 逻辑树中的旧节点，再 `INSERT` 新节点，严禁利用 Unique Index 做 UPSERT 绕过。事务内必须引入 `SELECT ... FOR UPDATE` 锁定目标行。物理文件的防重复必须通过底层的 `knowledge_physical_files.content_hash` 全局唯一索引硬性保障。

## 二、 网络路由与算力安全约束 (Network & AI Compute Security)

### 1. 算力节点内网穿透阻断
系统核心驱动为部署在局域网的 Ollama LLM 引擎 (`10.132.60.133`)。
* **隔离策略**：严禁前端代码发起对大模型节点的直连。
* **后端守卫与限流**：所有大模型请求必须通过后端的 FastAPI 服务进行代理，且后端必须配置 `Routing Guard`，仅放行来源为 `10.132.x.x` 或 `127.0.0.1` 的内部流量。同时，必须在后端网关层对 `POST /chat` 及 `POST /tasks/polish` 实施基于 Redis Token Bucket 的严格限流机制（如 5次/分钟/用户），拦截超出抛出 429。

### 2. 全局防泄密水印强制挂载
* 前端根视图必须强制挂载 `AntiLeakWatermark` 组件。该组件需动态提取当前用户的工号、所在科室名及当前时间，以 `0.08` 的极低透明度斜向覆盖全屏（`pointer-events: none`），确保所有截图行为均可被溯源定位。

## 三、 UI/UX 拟物与物理排版契约 (UI & Formatting Conventions)

### 1. 国标 A4 物理约束
前端公文编辑工作区（`EditorWorkspace.tsx`）**严禁**使用响应式宽度的 `width: 100%` 或流式布局。
* **尺寸锁定**：编辑器画板的基准物理宽度必须绝对锁定为 `794px`（A4 纸张的 96DPI 标准转换）。
* **动态防变形**：通过监听外部滚动容器宽度，动态计算并应用 `transform: scale()` 进行整体缩放。

### 2. 政务字体防御链
全局 CSS 变量中，针对公文正文区域的 `font-family` 设定必须严格按照以下降级顺序排列：
`font-family: "方正仿宋_GBK", "FZFS", "仿宋_GB2312", "仿宋", "FangSong", "STFangsong", "华文仿宋", "Noto Serif CJK SC", serif;`
必须将信创环境和政务内网的特定中文字体放在首位，严禁回退或使用系统默认的无衬线黑体（Sans-serif）渲染公文正文。

## 四、 鉴权隧道与防并发冲突契约 (Auth & Concurrency)

### 1. SSE 阅后即焚票据安全约束
原生 SSE 无法附加 `Authorization` Header，但**坚决禁止通过任何形态直接使用令牌做 URL 查询连接 (如：`?token=eyJ...`)。**
* **交换协定**：所有 SSE 长连接前置要求利用 Bearer Header 访问 `POST /api/v1/sse/ticket` 开具具备唯一且 TTL 少于 15 秒存活极限的票据。开具前，后端必须验证 `task_owner:{task_id}` 是否与当前用户匹配。
* **信道贯通**：前置换发获票后，经 `?ticket={Ticket}` 叩通任务引擎。后端必须在校验通过后即刻销毁 Ticket，确保阅后即焚。

### 2. Redlock 唯一真理源与断网容灾释放
* **前置审批状态网关**：锁路由必须阻隔非起草 (`DRAFTING`) 阶段文件的编辑态下探请求。
* **拒绝双规脑裂**：获取及控制并发必须以 Redis 的状态为 `Single Source of Truth`。
* **防丢失强兜底**：前端退出前，必须利用 `fetch` 搭配 `keepalive: true` 发射最后的自动保存与锁释放请求。

### 3. 锁的协作交互与超时释放
* **锁催办 (Nudge) 机制**：后端应提供 Nudge 接口，通过 SSE 隧道实现实时协作通知，防止编辑通道被长期霸占。
* **前端主动释放约束**：前端必须具备无输入自动释放机制（15分钟）。该动作必须由前端检测后主动调用 `/unlock` 接口完成，而非仅依赖 Redis TTL。释放前需确保通过 `keepalive` 发送最后一份自动保存草稿。

## 五、 全局防弹与容灾自愈 (Bulletproof & Recovery)

### 1. 前端报错防吞噬原则
在前端的请求拦截器中，所有 HTTP 4xx 和 5xx 的后端报错必须通过 Ant Design 的 `message.error` 原样透传并展示。

### 2. 全节点多轨漫游防灾
系统必须以云端 `document_snapshots` 快照为主存储的唯一灾难恢复受信任数据源。
* **云灾备系统**：对系统执行"应用AI排版重写"等高重写风险动作前，并行派发写入远端 `document_snapshots` 的请求。
