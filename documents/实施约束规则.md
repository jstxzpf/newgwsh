# 国家统计局泰兴调查队公文处理系统 V3.0 - 特殊约定与开发铁律

## 一、 数据库与会话隔离铁律 (Database & Session Isolation)

### 1. 严格的异步与同步物理隔离
由于系统在同一个代码库中混合了高并发 HTTP 网关（FastAPI）与重型后台计算（Celery），数据库会话的调用方式是不可逾越的红线：
* **路由层（API 层）**：必须且只能使用 `AsyncSessionLocal` (基于 `asyncpg` 引擎) 处理前端请求。
* **后台任务（Celery `worker.py`）**：必须且只能使用 `SyncSessionLocal` (基于 `psycopg2` 引擎)。**严禁在 Worker 中使用异步会话**，否则会导致协程事件循环直接崩溃。
* **Celery 模块注册规则**：在 `celery_app.py` 实例化时，必须显式通过 `include=["app.tasks.worker"]` 参数包含任务定义模块，严禁依赖自动扫描，确保 Worker 启动后任务能被正确注册。

### 2. 强制软删除与不可篡改日志
* 针对 `knowledge_base_hierarchy` (知识库)、`exemplar_documents` (范文库) 与核心公文表（`documents`），严禁在代码层执行 `session.delete()`。必须采用更新 `is_deleted = True` 的软删除策略，以符合国家统计局数据留存审计要求。
* 针对 `nbs_workflow_audit` (审计表) 和 `document_approval_logs` (审批存证表)，仅允许 `INSERT` 操作（Append-Only）。任何对其执行 `UPDATE` 或 `DELETE` 的 ORM 动作均属严重安全违规。
* **物理去重与切片复用契约**：在创建 `FILE` 节点前，必须先进行哈希碰撞检测。若命中且已有有效切片，**必须执行安全等级一致性校验**：若新节点的 `security_level` 高于现有切片，**严禁复用**，强制重新以高等级标签执行解析入库；否则可建立复用关联。
* **逻辑防线 (RAG)**：在执行向量检索或 BM25 查询时，**必须强制 `JOIN knowledge_base_hierarchy` 并过滤 `h.is_deleted = FALSE`**。
* **索引维护审计**：级联软删除波及的切片 ID 与内容哈希必须记入审计表。后端 Cron 任务必须定期清理 GIN 索引中的无效词条，防止索引膨胀。**执行置空清理时，为规避与正在执行的解析（PARSE）任务产生死锁竞态，必须在 UPDATE 语句的子查询中显式附加 `FOR UPDATE SKIP LOCKED` 机制**。

### 3. 状态机转换防线 (State Machine Guard)
公文状态机是系统最核心的业务约束，**必须在应用层和数据库层双重强制执行**，防止因 API 调用顺序错乱导致"无主编辑态"或"跨状态审批"等灾难性数据损毁。

* **应用层防线（ORM 钩子）**：在 SQLAlchemy 模型中，针对 `documents.status` 字段定义 `@validates('status')` 钩子，在任何 ORM 赋值操作前进行合法性校验：
  ```python
  VALID_TRANSITIONS = {
      'DRAFTING':  ['SUBMITTED'],
      'SUBMITTED': ['APPROVED', 'REJECTED'],
      'APPROVED':  [],           # 终态，拒绝一切转换
      'REJECTED':  ['DRAFTING'], # 仅允许回退起草
  }
  ```
  若转换路径不合法，直接抛出 `ValueError`，由全局异常处理器转换为 HTTP 409。
* **接口层前置校验（强制执行顺序）**：
  1. `POST /documents/{doc_id}/submit`：**第一步**必须校验 `status == DRAFTING`，否则返回 409
  2. `POST /approval/{doc_id}/review`：**第一步**必须校验 `status == SUBMITTED`，否则返回 409
  3. `POST /documents/{doc_id}/revise`：**步骤 0**（抢锁之前）必须校验 `status == REJECTED`，否则返回 409
* **数据库层防线（DB CHECK 约束）**：建议在 `documents` 表的 DDL 中增加 PostgreSQL `CHECK` 约束（或触发器），确保在应用层绕过 ORM 直接操作数据库时，依然无法写入非法状态值。

### 4. 结构化语义切片准则
* **语义锚点保护**：对于统计法规等结构化文档，切片算法必须识别标题层级（H1-H6）。每个切片块除正文外，必须在元数据中强制注入其所属的**完整标题路径锚点**。禁止出现将标题与直接从属条文物理切割在不同 Chunk 中的现象。
* **Tabular RAG 强制规范**：针对表格文件，必须识别首行表头。每个数据行切片必须包含该行的文本摘要描述（Row Summary），严禁仅存储原始 JSON，以确保语义召回的有效性。
* **压缩包目录还原契约**：处理 `.zip/.tar.gz` 时，解压逻辑必须深度遍历目录树。**后端必须递归创建对应的 `DIRECTORY` 逻辑节点，严禁将压缩包内文件平铺上传至根目录**。
* **解析引擎一致性**：切片引擎必须与物理排版所用的 Markdown AST 解析逻辑保持高度一致，**全线代码明确锁定唯一授权使用 `markdown-it-py` 解析库生成抽象语法树**，严禁模块私自引入多重异构引擎，确保"所见即所分"。

### 5. 审计与事务隔离规约
* **审计非阻塞性**：`nbs_workflow_audit` 的写入严禁置于业务逻辑的主事务中以防事务污染。必须使用异步任务（Celery）或在 `After-Commit` 数据库钩子中触发。注意：该表中包含驳回理由等敏感明文，其可见性仅限起草人和涉事审批人，超管等非涉事人员查阅日志必须通过授权系统并审计其查阅行为。
* **版本替换并发锁与防重机制**：执行知识库文件替换时，必须先 `UPDATE` 逻辑树中的旧节点，再 `INSERT` 新节点，严禁利用 Unique Index 做 UPSERT 绕过。事务内必须引入 `SELECT ... FOR UPDATE` 锁定目标行。物理文件的防重复必须通过底层的 `knowledge_physical_files.content_hash` 全局唯一索引硬性保障；而逻辑记录的防重则在业务层约束（`BASE`/`DEPT` 检测同层级，`PERSONAL` 依赖部分唯一约束：`CREATE UNIQUE INDEX idx_kb_personal_unique ON knowledge_base_hierarchy(physical_file_id, owner_id) WHERE kb_type='FILE' AND kb_tier='PERSONAL' AND is_deleted=FALSE;`）。替换时若新上传文件算出的哈希等同于旧物理文件，则拦截阻断（内容无变化）。

### 5.1 多步写入原子性铁律
* **显式事务强制包裹**：所有涉及"先备份后覆写"的多步数据库写入操作（如 `apply-polish` 的快照备份 → 正文覆写 → 字段清空），**必须包裹在显式数据库事务中**（异步层用 `async with session.begin():`，同步层用 `with sync_session.begin():`）。严禁在 SQLAlchemy 自动提交（`autocommit`）模式下分散执行多步写入，否则将产生部分成功的数据损毁状态。**原则**：若任何一步写入失败，整个事务必须回滚，确保操作的全有或全无（All-or-Nothing）语义。

### 5.2 全局鉴权 is_active 检查铁律
* **停用用户防护**：在所有涉及**审批、锁获取、知识库写入、范文库写入**等敏感操作的鉴权中间件与业务校验层，**必须全局强制叠加 `system_users.is_active == True` 的校验**。已停用（软删除）的用户账号，无论其 `role_level` 或 `dept_head_id` 绑定状态如何，一律返回 HTTP 403 并写入安全审计日志，防止历史账号被恶意利用或遗留访问权限造成安全漏洞。

### 6. 安全令牌与通讯指纹
* **令牌续期与心跳铁律**：前端 `useLockGuard` 必须使用 **Web Worker** 维持心跳以抵御浏览器节流。初始心跳间隔由 `GET /api/v1/locks/config` 接口下发（默认 90 秒）；此后每次心跳成功响应中的 `next_suggested_heartbeat` 字段将**动态覆盖**此间隔，前端以最新一次心跳响应为准，严禁固定步进。
* **SSE 权限锚定**：SSE Ticket 申请时必须在 Redis 中绑定 `task_id` 与 `user_id`，连接握手时必须校验该绑定关系。
* **签名一致性与归一化流水线**：SIP 指纹计算严格采用公式 `normalized_content | reviewer_user_id | reviewed_at_iso8601`。**归一化逻辑（NFKC + 换行归一 + 空白压缩 + 连续换行压缩）必须作为独立函数在各模块间严格复用**。若归一化后内容为空，审批通过时必须拒绝生成 SIP 签名，防止出现无意义的防伪签章。系统后续提供的校验接口必须按 `|` 分割并对转义字符进行还原，确保验证的一致性。

### 7. 范文库约束铁律
* **格式限定**：范文库（`exemplar_documents`）仅允许上传 `.docx` 格式文件。后端在接收文件前必须校验 MIME 类型为 `application/vnd.openxmlformats-officedocument.wordprocessingml.document`，严禁仅依赖文件扩展名校验（防止扩展名伪造）。
* **范文库不参与 RAG**：范文记录不得创建 `knowledge_chunks` 向量切片，严禁将其纳入 BM25 或 HNSW 向量检索管道。范文仅作为 AI 润色的结构性 `few-shot` 模板，通过 `exemplar_id` 在润色任务参数中独立传递。
* **权限两级制**：范文库仅设 `BASE`（全员只读，仅管理员可写）和 `DEPT`（科室内只读，仅科室负责人可写）两个层级。不设个人范文库（个人草稿已有快照机制）。
* **软删除保护**：范文删除必须采用软删除（`is_deleted = True`），且删除前须校验该范文是否被正在进行中的润色任务引用；若有，须先等待任务完成或取消任务，再执行删除。

### 8. 提示词外置与热加载规约
* **物理隔离**：所有 AI 提示词（System Prompt、few-shot 模板、词汇审查词表）必须存放在后端 `app/prompts/` 独立目录下，以 `.txt` 格式的纯文本文件管理。**严禁将提示词字符串硬编码在 Python 业务代码中。**
* **加载机制**：服务启动时，`PromptLoader` 单例负责读取全部文件并缓存至内存 Dict（key = 文件名去后缀）。提供 `get_prompt(name: str) -> str` 方法供业务调用。
* **热加载**：管理员可通过 `POST /api/v1/sys/reload-prompts` 触发重载，`PromptLoader` 重新扫描目录并更新缓存。操作写入审计日志。前端保存提示词前弹出二次确认。
* **占位符规范**：提示词中支持 `{context}`、`{exemplar_text}`、`{doc_type_name}` 等占位符，由调用方通过 Python `str.format_map()` 安全替换。严禁使用 `eval` 或 `exec` 动态执行提示词内容。

## 二、 网络路由与算力安全约束 (Network & AI Compute Security)

### 1. 算力节点内网穿透阻断
系统核心驱动为部署在局域网的 Ollama LLM 引擎，其连接地址通过环境变量 `OLLAMA_BASE_URL` 配置（开发环境默认为 `http://10.132.60.133:11434`，生产环境根据实际部署拓扑配置）。
* **隔离策略**：严禁前端代码发起对大模型节点的直连。
* **后端守卫与限流**：所有大模型请求必须通过后端的 FastAPI 服务进行代理，且后端必须配置 `Routing Guard`，仅放行来源为 `10.132.x.x` 或 `127.0.0.1` 的内部流量，防止跨网段的非法算力盗用与数据刺探。同时，必须在后端网关层对 `POST /chat` 及 `POST /tasks/polish` 实施基于 Redis Token Bucket 的严格限流机制（如 5次/分钟/用户），拦截超出抛出 429。

### 2. 全局防泄密水印强制挂载
* 前端根视图必须强制挂载 `AntiLeakWatermark` 组件。该组件需动态提取当前用户的工号（`username`）、真实姓名（`full_name`）、所在科室名及当前时间，以 `0.08` 的极低透明度斜向覆盖全屏（`pointer-events: none`），确保所有截图行为均可被溯源定位。**严禁将 `client_ip` 显示在水印中**，以免引发敏感隐私合规问题。

## 三、 UI/UX 拟物与物理排版契约 (UI & Formatting Conventions)

### 1. 国标 A4 物理约束
前端公文编辑工作区（`EditorWorkspace.tsx`）**严禁**使用响应式宽度的 `width: 100%` 或流式布局。
* **尺寸锁定**：编辑器画板的基准物理宽度必须绝对锁定为 `794px`（A4 纸张的 96DPI 标准转换）。
* **动态防变形**：通过监听外部滚动容器宽度，动态计算并应用 `transform: scale()` 进行整体缩放。以此保障国标 GB/T 9704-2023 中规定的**"单行 28 个全角字符"**的硬性排版要求不被破坏、不出现意外换行。

### 2. 政务字体防御链
全局 CSS 变量中，针对公文正文区域的 `font-family` 设定必须严格按照以下降级顺序排列：
`font-family: "方正仿宋_GBK", "FZFS", "仿宋_GB2312", "仿宋", "FangSong", "STFangsong", "华文仿宋", "Noto Serif CJK SC", serif;`
必须将信创环境和政务内网的特定中文字体放在首位，严禁回退或使用系统默认的无衬线黑体（Sans-serif）渲染公文正文。

### 3. 多文档类型排版约束
* **类型锁定时机**：公文在创建（`POST /documents/init`）时即需选择文档类型（`doc_type_id`），一旦提交审批（`SUBMITTED`），文档类型字段锁定，任何修改请求均返回 HTTP 409。
* **排版分支强制隔离**：后端排版 Worker 在接收 FORMAT 任务时，必须根据 `doc_type_id` 加载对应的要素规则配置，严禁使用同一套排版逻辑处理所有文种。
* **结构不匹配处理**：若正文结构与所选文种要素不匹配（如"请示"缺少结尾请批段落），后端排版完成后应在 `result_summary` 中附加结构警告列表，前端以黄色提示展示，**允许用户下载该产物但须明确知晓警告**。不强制拦截。
* **通用文档与统计文种免检**：当 `doc_type_id` 对应的 `type_code = GENERAL` 时，`layout_rules.required_sections` 为空数组，排版 Worker **跳过结构校验分支**，仅执行基础物理格式化（字体、行距 28pt、首行缩进、页边距），不输出结构警告。`RESEARCH`（调研分析）与 `ECONOMIC_INFO`（经济信息）文种走专业结构校验分支，结构不符时附加警告但不拦截下载。

## 四、 鉴权隧道与防并发冲突契约 (Auth & Concurrency)

### 1. SSE 阅后即焚票据安全约束
原生 SSE 无法附加 `Authorization` Header，但**坚决禁止通过任何形态直接使用令牌做 URL 查询连接 (如：`?token=eyJ...`)，一经发现直接定性为中高级生产漏洞拦截。**
* **交换协定**：所有 SSE 长连接前置要求利用 Bearer Header 访问 `POST /api/v1/sse/ticket` 开具具备唯一且 TTL 少于 15 秒存活极限的换领票根 UUID `Ticket` 并发往前端。**开具票据前，后端必须提取 `Bearer Token` 验证身份，比对 Redis 中存储的 `task_owner:{task_id}` 是否与当前用户匹配，不匹配则拒绝颁发。**
* **信道贯通**：前置换发获票后，经 `?ticket={Ticket}` 叩通对应任务引擎建立管道。握手解析一旦验证确认立刻清除缓存。后端在处理 Ticket 校验时，**必须且只能执行一次性销毁动作（DEL key）**，确保 Ticket 在校验通过后即刻失效，彻底阻断同一 Ticket 的重播利用。

### 2. Redlock 唯一真理源与断网容灾释放
* **前置审批状态网关**：触发争抢与编辑前，锁路由必须阻隔非起草 (`DRAFTING`) 阶段文件的编辑态下探请求。
* **拒绝双规脑裂**：获取及控制该阶段并发必须以 Redis 的状态为 `Single Source of Truth`。**在执行 `revise` 操作进行状态回退时，必须在同一个后端事务内通过 Redis 的 `SET lock:{doc_id} token NX EX 180` 原子命令强制抢锁。**
  - 若抢占失败（409）：告知锁已被抢占，不再颁发 Token，状态不回退。
  - 若抢占成功：自动为该用户颁发锁 Token 并回退状态。
  此原子性设计彻底根除了获取锁与状态变更间的竞态条件。前端只需一次请求即可带锁进入工作区。
* **防丢失强兜底**：前端退出或者卸载销毁前，不得单独发送释放锁通讯。强制将目前缓存最新的业务实体封装塞入 payload 向心跳通道抛掷并同时命令放弃持锁（**必须利用 `fetch` 搭配 `keepalive: true` 属性**执行，严禁使用无法携带自定义 Header 的 `sendBeacon`），以此防止自动间隔导致的边缘工作损失。

## 五、 全局防弹与容灾自愈 (Bulletproof & Recovery)

### 1. 前端报错防吞噬原则
在前端的请求拦截器（`apiClient`）或具体的业务 Try-Catch 块中，**严禁**出现仅使用 `console.log(error)` 的静默吞没行为。
* 除 `401 Unauthorized`（统一拦截并交由全局路由至 `/login`）外，所有 HTTP 4xx 和 5xx 的后端报错（尤其是后端的 `detail` 字段），必须通过 Ant Design 的 `message.error` 原样透传并弹窗展示给用户。

### 2. 全节点多轨漫游防灾
公文必须防患多端切换编辑的不安全隐患，拒绝以客户端本地 `Local Storage` 成为灾难备份的全部支撑点。**需特别明确边界**：本地缓存池内仅存留最近 2 条记录，这属于解决编辑器崩溃的极微秒级临时状态极速恢复机制，绝对不属于灾难备份策略，系统仍且必须以云端 `document_snapshots` 快照为主存储的唯一灾难恢复受信任数据源。
* **云灾备系统**：对系统执行"应用AI排版重写"等单边高重写风险动作前，并行派发一条写入远端数据图谱的 `POST` 请求（`document_snapshots`）。即使客户端粉碎换台仍可在侧边栏拉取复旧容灾操作。
* **漫游 DIFF 识别**：若异地调取文档发现其含有后生成未决的 `AI` 同步段落值，强制复原渲染成对比 DIFF 控制板，摆脱客户端 Local 的状态绑架。

## 六、 规范化运维与可观测性守护 (Monitoring & SOP)

系统的状态不能处于盲测之中。后端代码合入主干前必须满足监控检查清单：
* **结构化日志**：各层（如 `api`, `services`, `tasks` 等）关键出口必须部署基于 `structlog` 渲染的标准化 JSON 日志，包涵发生时刻、模块节点及 `doc_id`/`task_id`，并必须携带全局唯一的 `trace_id` 将孤立环节串联为全链路流。
* **高危操作探针**：涉及 `审批判决`、`文档清除` 及 `权限变动` 的接口必须显式吐出可采集的 `INFO` 或 `WARNING` 日志以便审计系统抓取。
* **算力熔断告警**：连接 Redis 及 Ollama 的服务句柄应当配置自动超时；Celery 死信队列及 `async_tasks` 中的滞留 FAILED 任务过载（如 5 分钟内错误超标）时，需抛出含 Traceback 的 `CRITICAL` 告警。

## 七、 状态机与并发安全铁律 (State Machine & Concurrency)

### 1. 原子性回退与抢锁顺序 (P0)
在实现 `POST /documents/{doc_id}/revise` 接口时，**必须首先执行前置状态校验**（`status == REJECTED`，否则 HTTP 409），**其次执行 Redis 抢锁动作**（`SET lock:{doc_id} token NX EX 180`）。只有抢锁成功（获得 Lock Token）后，才能在数据库事务中将公文状态从 `REJECTED` 回退为 `DRAFTING`。严禁先回退状态后抢锁，以防状态回退后因抢锁失败导致的"无主编辑态"或死锁。

### 2. 前端容灾释放规范 (P2)
在 `window.beforeunload` 事件中释放 Redis 锁时，**严禁使用 `navigator.sendBeacon`**。必须使用 `fetch` 接口并设置 `keepalive: true`，以确保能够携带 JWT 认证 Header 并成功释放受保护的后端接口锁。所有锁相关操作必须包含前端生成的 `lock_token` 进行凭证比对。
