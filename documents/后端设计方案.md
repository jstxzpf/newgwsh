# 国家统计局泰兴调查队公文处理系统 V3.0 - 后端设计方案

## 一、 后端架构分层与职责隔离 (Architecture & Layers)

后端基于 **FastAPI + Celery + PostgreSQL/Redis** 搭建高并发响应与重型计算分离的架构体系。

> **对 AI Agent 的架构约束指令**：
> 1. **严禁在 FastAPI 主线程中执行耗时阻塞操作**（如大模型推理、Word 排版、长文本解析），必须通过 `task.delay()` 压入 Celery。
> 2. **数据库引擎必须物理隔离**：在 `backend/app/core/database.py` 中，必须定义 `AsyncSessionLocal`（基于 asyncpg，供 FastAPI 路由使用）和 `SyncSessionLocal`（基于 psycopg2，专供 Celery 的 `worker.py` 使用）。严禁在 Celery 任务中使用 AsyncSession，否则会导致协程事件循环崩溃。

### 核心目录映射
* `app/api/v1/`: HTTP 路由网关（`tasks.py`, `chat.py`, `kb_admin.py`, `locks.py`, `approval.py`, `documents.py`）
* `app/core/`: 核心基建（`config.py`, `database.py`, `models.py`, `security.py`）
* `app/services/`: 业务逻辑抽象（如 `docx_service.py` 负责物理排版，`draft_service.py` 负责草稿自动保存）
* `app/tasks/`: 异步后台服务（`worker.py` 包含所有的 Celery Task）
* `app/prompts/`: **AI 提示词模板目录**。所有 System Prompt、few-shot 模板和词汇审查词表均以独立文件存* **步骤 1：任务派发**。FastAPI 接收前端润色/排版请求（如润色任务须接收 `doc_id`, `context_kb_ids`, `context_snapshot_version`, `exemplar_id` 等参数），生成全局唯一 `task_id`，将初始状态写入 Redis (`idempotency:task:{task_id}`) 并同步写入 `async_tasks` 持久化表，**同时强制在 Redis 写入 `task_owner:{task_id} = user_id`（TTL 与任务对齐）**，随后调用 Celery 的 `.delay()`。接口立即返回 HTTP 202 及 `task_id`。
* **步骤 2：换取 Ticket 并建立隧道**。前端拿到 `task_id` 后，由全局组件 `GlobalTaskWatcher` 统一管理。首先访问 `POST /api/v1/sse/ticket` 申请短期临时通讯票据（TTL 15秒）。**该接口必须前置校验 Redis 中的 `task_owner:{task_id}` 是否等于当前请求的 `user_id`**。通过后，前端利用 `EventSource` 访问 `/api/v1/sse/{task_id}/events?ticket={Ticket}`。后端校验 Ticket 并阅后即焚。
* **步骤 3：状态接力**。Celery Worker 在后台执行任务。**对于 `POLISH` 任务，Worker 必须首先检查透传的 `exemplar_id` 对应的范文是否被软删除（`is_deleted=True`）。若范文已失效，则任务自动降级为“无范文”模式继续执行，并在最终的 `result_summary` 中记录“范文已失效降级”。** 当进度到达 100% 时，Worker 必须将结果文本写入 `documents.ai_polished_content`；若为 `FORMAT` 任务，则生成 Word 文件并写入 `documents.word_output_path`。随后更新 Redis 状态为 `COMPLETED`。
* **步骤 4：闭环推送与全局通知**。SSE 隧道实时推送进度百分比。当前端收到 `COMPLETED` 事件时，右下角弹出全局 Notification 通知卡片（文案见 UI 设计），提供“查看详情”和“忽略”按钮。若用户点击详情，则跳转至对应工作区并启动 DIFF 模式。送进度，完成后前端根据此事件刷新库面）

* **步骤 1：任务派发**。FastAPI 接收前端润色/排版请求，生成全局唯一 `task_id`，将初始状态写入 Redis (`idempotency:task:{task_id}`) 并同步写入 `async_tasks` 持久化表，**同时强制在 Redis 写入 `task_owner:{task_id} = user_id`（TTL 与任务对齐）**，随后调用 Celery 的 `.delay()`。接口立即返回 HTTP 202 及 `task_id`。
* **步骤 2：换取 Ticket 并建立隧道**。前端拿到 `task_id` 后，由全局组件 `GlobalTaskWatcher` 统一管理。首先访问 `POST /api/v1/sse/ticket` 申请短期临时通讯票据（TTL 15秒）。**该接口必须前置校验 Redis 中的 `task_owner:{task_id}` 是否等于当前请求的 `user_id`**。通过后，前端利用 `EventSource` 访问 `/api/v1/sse/{task_id}/events?ticket={Ticket}`。后端校验 Ticket 并阅后即焚。
* **步骤 3：状态接力**。Celery Worker 在后台执行任务。**当任务类型为 `POLISH` 且进度到达 100% 时，Worker 必须将结果文本写入 `documents.ai_polished_content`；若为 `FORMAT` 任务，则生成 Word 文件并写入 `documents.word_output_path`**。随后更新 Redis 状态为 `COMPLETED`。
* **步骤 4：闭环推送与全局通知**。SSE 隧道实时推送进度百分比。当前端收到 `COMPLETED` 事件时，右下角弹出全局 Notification 通知卡片（文案见 UI 设计），提供“查看详情”和“忽略”按钮。若用户点击详情，则跳转至对应工作区并启动 DIFF 模式。
* **步骤 5：失败推送与补偿**。若任务抛出异常，状态置为 `FAILED`。SSE 推送失败事件，前端提示重试。若 SSE 断连，前端通过 `GET /api/v1/tasks/{task_id}` 进行状态补偿查询。

### 2. 草稿自动保存流 (Auto-Save Pipeline)
为防止用户编辑内容因浏览器崩溃或网络中断而丢失，后端提供主动拉取式的草稿自动保存机制。

* **前端定时触发**：前端每 60 秒检测变化。**接口路径调整为 `POST /api/v1/documents/{doc_id}/auto-save`**，以兼容 `navigator.sendBeacon` 与 `fetch` `keepalive` 机制。
* **后端幂等写入与 DIFF 保护**：后端接收后，自行计算前端传来的正文内容的 SHA-256 哈希值，若与数据库当前内容一致则直接返回 HTTP 200 避免无意义写入。
  **自动保存与 DIFF 保护状态处理矩阵**：
  为防止 DIFF 模式下的建议稿与原稿混淆覆盖，后端严格遵循以下键值接收规则：
  | 当前公文状态 | 请求载荷 (Payload) 组合 | 后端处理动作 |
  | :--- | :--- | :--- |
  | **DIFF 模式**<br/>(`ai_polished_content` 不为空) | 仅包含 `draft_content` | **允许**。更新至 `draft_suggestion` 字段，保护二改进度。绝对不更新 `content`。 |
  | **DIFF 模式** | 包含 `content` 键 (不论其值为何) | **拒绝 (HTTP 400)**。防止绕过保护机制直接覆写原稿。 |
  | **SINGLE 模式**<br/>(`ai_polished_content` 为空) | 仅包含 `content` | **允许**。正常更新至 `content` 字段。 |
  | **SINGLE 模式** | 包含 `draft_content` 键 | **拒绝 (HTTP 400)** 或 **直接忽略该键**。状态不一致防误伤。 |
* **状态不变性**：自动保存不触发任何工作流节点变更。

### 3. 公文并发防篡改与 SIP 存证流
针对多人同处一室编辑同一份台账的冲突场景，以及最终的司法合规要求。

* **悲观锁抢占 (Redlock)**：前端进入起草模式前，系统必须前置校验 `document.status == DRAFTING`（非草稿态一律拒绝加锁申请）。校验通过后向 `/locks/acquire` 申请该 `doc_id` 的排他锁。后端**仅在 Redis 中**设置带 TTL（如 3 分钟）的锁。锁 Key 结构为 `lock:{doc_id}`，Value 存储 JSON 结构：`{"user_id": 123, "username": "xxx", "acquired_at": "ISO8601", "token": "uuid-v4"}`（**注意必须包含生成的凭证 token**）。若获取失败则抛出 HTTP 409，前端进入只读模式。后续心跳与释放均必须校验前端传入的 `lock_token` 与 Redis 内 `token` 一致。
* **锁的完整释放策略**：
  * **主动释放**：用户离开编辑页面时，前端在 `useEffect cleanup` 中调用 `POST /locks/release`。后端必须校验 `current_user.user_id == lock_holder.user_id` 且 `token` 一致，否则拒绝释放。
  * **容灾释放**：前端注册 `window.beforeunload` 事件，优先发射最后一次未保存的正文到 `/auto-save`，通过 `fetch` 搭配 `keepalive: true` 向 `/locks/release` 发送包含 Header 的释放请求，确保数据不丢失。
  * **提交验证与释放**：后端在 `POST /documents/{doc_id}/submit` 时，**第一步必须校验 `documents.status == DRAFTING`**，非起草态（如已 `SUBMITTED` 或 `APPROVED`）一律返回 HTTP 409，防止重复提交。校验通过后处理锁逻辑：如果锁 Key 存在，必须校验锁 Key 中的 `user_id` 与当前提交者一致，一致则允许提交并主动删除锁，不一致抛出 409；如果锁 Key 不存在（可能已 TTL 过期），则校验当前提交者必须是该公文的起草人（`creator_id`），若一致仍允许提交（容忍过期），否则抛出 403。**提交成功后须将 `submitted_at` 写入 `document_approval_logs` 对应记录（创建待审批记录时），并在 HTTP 响应中返回 `{"doc_id": "...", "status": "SUBMITTED", "log_id": 123}`。**
  * **延时预占与心跳续期**：公文起草阶段，仅在产生输入行为时通过 `POST /locks/acquire` 申请 Redis 锁（TTL 180s）。
  * **高精度续期校准**：前端心跳接口 `POST /locks/heartbeat` 的响应体必须返回 `next_suggested_heartbeat`（下次建议续期时间戳）及 `lock_ttl_remaining`（剩余秒数）。后端根据服务器负载动态调整建议间隔，前端依据此值进行计时校准。
  * **TTL 兜底与强拆**：Redis 自动清理过期锁。系统提供 `DELETE /locks/{lock_key}` API 供管理员强制夺锁，该操作必须写入审计日志。
  * **参数外置配置**：锁的基础 TTL 时长（默认 180 秒）和心跳间隔（默认 90 秒）必须抽取至后端的 `config.py` 环境变量中。配置将由 `GET /api/v1/locks/config` 接口下发。
* **SIP 哈希存证与归一化流水线**：科长在签批通过时，后端必须执行标准化的文本归一化流水线。
  ```python
  def normalize_for_sip(text: str) -> str:
      import unicodedata, re
      # 1. NFKC 兼容性等价分解
      text = unicodedata.normalize('NFKC', text)
      # 2. 换行符归一化 (\r\n -> \n)
      text = text.replace('\r\n', '\n').replace('\r', '\n')
      # 3. 对存证分隔符进行转义
      text = text.replace('|', '\\|')
      # 4. 压缩连续的非换行空白符为单个空格
      text = re.sub(r'[^\S\n]+', ' ', text)
      # 5. 压缩连续换行为单换行
      text = re.sub(r'\n+', '\n', text)
      # 6. 首尾去除
      text = text.strip()
      return text
  ```
  做完归一化处理后，使用 `SIP_SECRET_KEY` 执行 **HMAC-SHA256** 生成指纹。
  **存证原串公式**：`normalized_content | reviewer_user_id | reviewed_at_iso8601`（其中 `reviewed_at` 的时间源必须取自 `document_approval_logs` 中该公文最后一条 `decision_status='APPROVED'` 记录的归档时间）。若归一化后内容为空，则拒绝签批。系统提供 `GET /documents/{doc_id}/verify-sip` 校验接口。

### 4. 签批驳回与回退流
* **驳回操作**：科长在 `/approval/{doc_id}/review` 接口选择"驳回"时，**必须提供 `rejection_reason`**（后端校验非空），系统执行：
  1. 在 `document_approval_logs` 表插入 `decision_status = REJECTED` 的新记录，含驳回理由。注意：`APPROVED` 的终态记录有且仅有一条，每次提交都会产生新的待审或流转记录。
  2. 在 `nbs_workflow_audit` 审计表注入 `REJECTED(41)` 节点，`action_details` 包含驳回理由 JSON。
  3. 将 `documents.status` 更新为 `REJECTED`。
  4. 通过 SSE 或通知机制向起草人推送驳回消息。
* **回退编辑**：起草人通过 `POST /documents/{doc_id}/revise` 接口。
  1. **步骤 0（前置状态校验）**：后端**首先**校验 `documents.status == REJECTED`。若当前状态非 `REJECTED`，立即返回 HTTP 409 "公文当前状态不允许回退修改"。
  2. **原子性抢锁**：校验通过后，使用 Redis 的 `SET lock:{doc_id} token NX EX 180` 命令抢占锁。若抢占失败则立即抛出 HTTP 409（此时状态不回退）。
  2. **状态回退**：抢占成功后，在同一个数据库事务内：将 `documents.status` 更新为 `DRAFTING`，同步清空 `ai_polished_content` 和 `draft_suggestion` 以去除旧版本污染。
  3. **审计与通知**：审计表注入 `REVISION(42)` 节点。系统主动向该锁的前一个持有者推送“锁已被回收”通知。
  4. **返回凭证**：为当前用户颁发并返回有效的锁 Ticket。响应结构：
  ```json
  {
    "doc_id": "uuid",
    "new_status": "DRAFTING",
    "lock_acquired": true,
    "lock_token": "uuid-token",
    "lock_expires_at": "ISO8601"
  }
  ```
  前端只需一次请求即可带锁进入工作区。

## 三、 核心算法设计 (Core Algorithms)

### 1. 混合检索算法 (HRAG: Hybrid Retrieval-Augmented Generation)
单纯的向量检索在面临"查询2024年一季度第一产业总产值"这种强精确数值词时极易失效，必须采用双路召回与融合打分。

> **对 AI Agent 的算法指令**：在 `chat.py` 混合检索算法中，**无论 HNSW 还是 BM25 计算，在最底层 SQL 查询中必须强制 JOIN `knowledge_base_hierarchy` 并共同过滤 `h.is_deleted = FALSE` 与 `knowledge_chunks.is_deleted = FALSE`**，严禁仅依赖单表字段导致目录级联删除穿透幽灵泄露。同时强制叠加底层权限条件过滤：对于 `kb_tier=BASE` 允许全部访问；`DEPT` 限制同科室成员（并在底层查询加上 `knowledge_chunks.dept_id` 匹配或 NULL 的条件以加速鉴权过滤）；`PERSONAL` 仅限 `owner_id = current_user.user_id`。

* **路经 A（语义向量召回）**：
  提取用户 Query 的 Embedding，使用 `pgvector` 的 `<=>` (Cosine Distance) 算子去 `knowledge_chunks` 表中查出 Top-K。
* **路经 B（BM25 全文精准召回）**：
  使用 PostgreSQL 的全文检索函数。为保障良好的中文搜索词法支持，必须为数据库安装 `pg_jieba` 或 `zhparser` 扩展。随后利用 `to_tsquery` 和 `ts_rank_cd` 等函数结合特定的词典进行查询，在同一个切片表中按词频查出 Top-K，保证索引建立阶段与查询阶段中文字典配置的强一致。
* **RRF (Reciprocal Rank Fusion) 重排序融合**：
  计算倒数排名融合得分：`RRF_Score = 1 / (k + Rank_Vector) + 1 / (k + Rank_BM25)`（k 常设为 60）。取融合后的前 3~5 个 Chunk，并在拼接给大模型之前二次遍历排除任何落库时 `security_level='CORE'` 的高绝密召回切片（将其静默废弃并入系统安全审计日志）。增加一个后置校验兜底：若 BM25 路返回候选数为 0 且向量路最高相似度低于阈值（如 0.7），则直接触发“未探明线索”话术，不强行给 LLM 拼接低质量上下文。之后将其拼接为最终的 LLM 提示词上下文。
* **防幻觉系统提示词（System Prompt）装配要求**：
  必须通过加载 `app/prompts/system_chat.txt` 中定义的系统角色设定模板植入，**严禁将提示词硬编码在业务代码中**。提示词文件支持 `{context}` 等占位符，由服务层在运行时通过 `str.format_map()` 安全替换。参考默认内容：
  _“你是一个严肃政务问答助手。请严格且只能依据下方【挂载上下文】的数据进行总结回答。若提供的信息不足以响应指令，请直接明示‘未探明对应线索’，严禁依靠自身推理或幻觉编造任何指标流或数据信息。”_

### 2. 知识库切片与解析流水线 (Chunking & Parsing Pipeline)

为确保“层次分明”与“切片复用”，解析流程执行以下技术标准：

*   **物理去重与切片复用契约**：
    1.  **哈希预检**：上传文件时计算 SHA-256 哈希。
    2.  **引用判断**：若物理记录已存在，执行**安全等级一致性校验**：若新节点的 `security_level` 高于现有切片，**必须拒绝复用**，强制重新以高等级标签执行解析入库；否则直接复用。
    3.  **切片复用**：若校验通过且已有 `parse_status = READY` 的切片，逻辑节点直接继承，状态秒变 `READY`。
*   **语义切片算法 (AST Semantic Chunking)**：
    *   **统一引擎**：强制使用 `markdown-it-py` 解析 Markdown AST。
    *   **标题路径注入**：每个切片必须在 `metadata_json` 中包含自根目录起的完整路径名（如：`{"title_path": "全局库/统计法/第一章"}`）。
    *   **分块边界**：以 H1-H6 标题为自然边界。仅当逻辑块超出 800 字符时，在段落间执行次级切割，严禁在句子中强制切断。
*   **表格增强解析 (Tabular RAG)**：
    *   针对 `.xlsx/.csv`，解析器需识别第一行为表头。
    *   **行摘要生成**：为每行数据生成描述性文本（例如：“2024年一季度工业产值数据，指标：总产值，数值：123.4亿元”）。
    *   **元数据关联**：切片需保留原始行列 JSON，以便 RAG 召回时支持精确数据比对。
*   **级联软删除技术 (Recursive CTE Cleanup)**：
    *   当删除 `DIRECTORY` 类型的节点时，后端必须使用 `WITH RECURSIVE` 递归查找所有层级的子节点。
    *   **事务一致性**：在一个事务内，将所有波及节点的 `is_deleted` 设为 `TRUE`，并同步更新 `knowledge_chunks` 表，将对应 `embedding` 置为 `NULL`。
    *   **索引维护与审计**：软删除波及的切片 ID 与内容哈希必须记入 `nbs_workflow_audit`。后端设置定期任务（Cron），清理 GIN 索引中已删除切片的无效词条，防止索引膨胀。**GIN 索引清理实现规范（采用方案B）**：PostgreSQL 的 GIN 索引仅在 `VACUUM` 时清理物理死元组，对 `is_deleted=TRUE` 但未物理删除的行中的词条无法自动回收。后端 Cron 任务须定期执行软删除文本清理，**为了防止与正在执行的 PARSE 解析任务抢占行锁发生死锁竞态，必须使用 `SKIP LOCKED` 机制**。推荐的 SQL 结构为：`UPDATE knowledge_chunks SET content = '' WHERE chunk_id IN (SELECT chunk_id FROM knowledge_chunks WHERE is_deleted = TRUE AND content != '' LIMIT 5000 FOR UPDATE SKIP LOCKED)`。建议频率：每日低峰期执行，避免写入风暴。

### 3. GB/T 9704-2023 政务公文物理排版算法
* **文种分支路由**：Worker 接收 FORMAT 任务后，**首先**根据 `doc_type_id` 加载 `document_types.layout_rules`。若 `required_sections` 为空数组（即 `type_code=GENERAL`），直接进入**基础格式化分支**：仅执行字体、行距 28pt、首行缩进 2 字符、页边距等物理格式化，跳过结构校验与结尾语注入，`result_summary` 中标记 `"mode": "basic_format"`，不生成结构缺失警告。`RESEARCH` 与 `ECONOMIC_INFO` 文种走**专业结构校验分支**：按其 `required_sections` 校验，结构不符则附加警告列表但允许下载。
* **词库清洗拦截**：在交由 `python-docx` 渲染前，从 `app/prompts/vocab_blacklist.txt` 加载屏蔽词表（每行格式：`原词→替换词`，运行时缓存，支持热加载），对传入的 `content` 运行正则替换（如"隆重召开"→"召开"、"亲自指示"→"指示"）。**所有文种（含 GENERAL）均执行此步骤。**
* **公文内容排重**：在生成 Word 前，执行双重排重：
  1. **段落哈希比对**：提取关键段落哈希，与同科室已审批公文比对（阈值 70%）。
  2. **向量语义比对**：增加基于 `documents.content` 归一化后文本整体嵌入向量的余弦相似度比对。若相似度 > 0.85 且哈希匹配度低，归类为“疑似改写”，向用户发出雷同预警并要求复核。
* **两办规范语料库校验**：增加对“禁止搭配不当”、“句子成分残缺”、“逻辑跳跃”的离线规则扫描（不依赖大模型），在排版前给出语法规范性建议。
* **基于 AST 的结构树遍历**：在物理排版解析阶段保持解析引擎底层依赖一致性，同样使用 `markdown-it-py` 将 Markdown 文本解析为抽象语法树，遍历节点：
  * **一级标题 (##)**：设置字体为"黑体"，三号，无加粗。
  * **二级标题 (###)**：设置字体为"楷体_GB2312"，三号。
  * **正文**：设置字体为"仿宋_GB2312"，三号，首行缩进 2 字符，行距固定值 28 磅。
  * 产物生成在 `/app/data/outputs` 目录下，并返回下载 URL。

   尽管应用层具备快照容灾机制，底层的数据持久化仍需依赖数据库原生备份。服务器必须配置 `pg_dump` 每天定时全量备份与基于 Write-Ahead Logging (WAL) 的归档策略。结合 `pg_basebackup` 确保在发生彻底硬盘损坏或被恶意软件勒索等灾难性故障时，能够执行 Point-in-Time Recovery (PITR) 恢复到具体发生时间点。

## 六、 日志与监控规范 (Logging & Monitoring)

### 1. 结构化日志标准
后端所有模块必须使用 Python `logging` + `structlog` 输出 JSON 格式结构化日志，包含以下必备字段：
* `timestamp`：ISO 8601 格式
* `level`：`INFO` / `WARNING` / `ERROR` / `CRITICAL`
* `module`：来源模块（如 `api.tasks`, `worker.polish`）
* `user_id`：操作人 ID（如有）
* `doc_id` / `task_id`：关联的业务实体 ID（如有）
* `trace_id`：全局请求链路追踪 ID（基于中间件全局拦截器在入口主动生成唯一 `request_id`，并向下贯穿下放至 Celery 的 Task kwargs 及底层日志链中，实现全量生命周期的串联组装关联查找，SSE 推流响应亦需携带此 ID）
* `message`：日志正文

### 2. 关键操作审计日志
以下操作必须输出 `INFO` 级别日志，并同步写入 `nbs_workflow_audit` 表：
* 公文创建/提交/签批/驳回
* 知识库上传/删除/替换
* 用户登录/登出
* 锁的获取/释放/超时

### 3. 异常告警与重试退避机制
* Celery Worker 进程退出或长时间无响应时，通过标准错误输出 `CRITICAL` 级别日志。
* **任务重试联动 (Retry Bound)**：在 Celery 任务装饰器显式设定 `max_retries=3` 和 `autoretry_for=(Exception,)`。任务内部捕获异常后，首要操作是更新 `async_tasks.retry_count += 1`。若 `retry_count <= 3`，调用 `self.retry()` 触发重试并**结束当前函数执行**；若 `retry_count > 3`，将状态置为 `FAILED`，存储错误堆栈，并不再抛出异常（阻断 Celery 内部重试接管）。这确保计数与重试严格一致。
* 环境参数必须在 `config.py` 中明确落定并暴露：Redis 连接超时降级为 5s、数据读写超时压控为 3s；Ollama HTTP 请求阻塞阀定格 120s。且 Celery 异步任务池必须硬防死锁滞留，装配 `soft_time_limit=600` 软熔断及 `time_limit=900` 线程斩断强裁决策机制。并将安全归档冷存储根路径规划给 `ARCHIVE_ROOT`。
* 建议对接日志收集系统（如 ELK 或 Loki + Grafana），并配置以下告警规则：
  * 5 分钟内 `ERROR` 日志超过 10 条
  * `async_tasks` 表中 `FAILED` 任务占比超过 20%
  * Redis 连接断开超过 30 秒
