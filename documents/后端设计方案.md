# 泰兴市国家统计局公文处理系统 V3.0 - 后端设计方案

## 一、 后端架构分层与职责隔离 (Architecture & Layers)

后端基于 **FastAPI + Celery + PostgreSQL/Redis** 搭建高并发响应与重型计算分离的架构体系。

> **对 AI Agent 的架构约束指令**：
> 1. **严禁在 FastAPI 主线程中执行耗时阻塞操作**（如大模型推理、Word 排版、长文本解析），必须通过 `task.delay()` 压入 Celery。
> 2. **数据库引擎必须物理隔离**：在 `backend/app/core/database.py` 中，必须定义 `AsyncSessionLocal`（基于 asyncpg，供 FastAPI 路由使用）和 `SyncSessionLocal`（基于 psycopg2，专供 Celery 的 `worker.py` 使用）。严禁在 Celery 任务中使用 AsyncSession，否则会导致协程事件循环崩溃。

### 核心目录映射
* `app/api/v1/`: HTTP 路由网关（`tasks.py`, `chat.py`, `kb_admin.py`, `locks.py`, `approval.py`, `documents.py`）
* `app/core/`: 核心基建（`config.py`, `database.py`, `models.py`, `security.py`）
* `app/services/`: 业务逻辑抽象（如 `docx_service.py` 负责物理排版，`draft_service.py` 负责草稿自动保存）
* `app/tasks/`: 异步后台服务（`worker.py` 包含所有的 Celery Task）

## 二、 核心工作流程解析 (Core Workflows)

### 1. 无阻塞长链接任务推流 (SSE Pipeline)
为解决大模型推理动辄几十秒导致的 HTTP 504 超时断连问题，采用基于 Redis 状态中转的 SSE 隧道机制。（注：`task_type=PARSE` 的知识库切片解析任务同样以此隧道推送进度，完成后前端根据此事件刷新库面）

* **步骤 1：任务派发**。FastAPI 接收前端润色/排版请求，生成全局唯一 `task_id`，将初始状态写入 Redis (`idempotency:task:{task_id}`) 并同步写入 `async_tasks` 持久化表，**同时强制在 Redis 写入 `task_owner:{task_id} = user_id`（TTL 与任务对齐）**，随后调用 Celery 的 `.delay()`。接口立即返回 HTTP 202 及 `task_id`。
* **步骤 2：换取 Ticket 并建立隧道**。前端拿到 `task_id` 后，先从 `POST /api/v1/sse/ticket` 接口申请短期临时通讯票据（此时在 Header 携带 Token）。**该接口必须前置校验 Redis 中的 `task_owner:{task_id}` 是否等于当前请求的 `user_id`**，若不匹配直接拦截。校验通过后，后端下发储存在 Redis 且 TTL 仅 15 秒的 Ticket。然后前端利用 `EventSource` 访问 `/api/v1/sse/{task_id}/events?ticket={Ticket}`。后端查验 Ticket 提取身份后立即阅后即焚，随后进入异步生成器循环。
* **步骤 3：多维通知频道**。SSE 隧道不仅推送任务进度，还承担**锁催办 (Nudge)** 消息的下行分发。
* **步骤 4：闭环推送与兜底**。FastAPI 的 SSE 循环每 0.5 秒轮询一次 Redis。当前端收到 `COMPLETED` 后重新拉取公文详情。若 SSE 断连，前端看门狗必须通过 `GET /api/v1/tasks/{task_id}` 接口进行状态补偿查询。
* **步骤 5：失败推送**。若 Worker 执行过程中抛出异常，Celery 的 `on_failure` 钩子捕获后将状态置为 `FAILED`。SSE 推送失败事件。前端通过重试接口触发任务重启。

### 2. 草稿自动保存流 (Auto-Save Pipeline)
为防止用户编辑内容因浏览器崩溃或网络中断而丢失，后端提供主动拉取式的草稿自动保存机制。

* **前端定时触发**：前端每 60 秒检测变化。**接口路径调整为 `POST /api/v1/documents/{doc_id}/auto-save`**，以兼容 `navigator.sendBeacon` 与 `fetch` `keepalive` 机制。
* **后端幂等写入与 DIFF 保护**：后端接收后，自行计算前端传来的 `content` 的 SHA-256 哈希值，若与数据库当前内容一致则直接返回 `{"changed": false, "saved_at": null}` (HTTP 200) 避免无意义写入（前端无需在 payload 中携带 hash 字段）。若内容变化再更新 `documents.content`。**关键防御**：若发现 `documents.ai_polished_content IS NOT NULL`（正处于 DIFF 模式），前端调用 `POST /auto-save` 时，请求体应为 `{"draft_content": "建议稿内容"}`。后端此时**必须拒绝任何包含 `content` 键的请求**并抛出 HTTP 400（防绕过直接覆写原稿）。接收到 `draft_content` 后，将其写入 `documents.draft_suggestion` 字段，且**绝对不更新 `content`**；反之，若不在 DIFF 模式，则仅接收并更新 `content`。
* **状态不变性**：自动保存不触发任何工作流节点变更。

### 3. 公文并发防篡改与 SIP 存证流
针对多人同处一室编辑同一份台账的冲突场景，以及最终的司法合规要求。

* **悲观锁抢占 (Redlock)**：前端进入起草模式前，系统必须前置校验 `document.status == DRAFTING`（非草稿态一律拒绝加锁申请）。校验通过后向 `/locks/acquire` 申请该 `doc_id` 的排他锁。后端**仅在 Redis 中**设置带 TTL（如 3 分钟）的锁。锁 Key 结构为 `lock:{doc_id}`，Value 存储 JSON 结构：`{"user_id": 123, "username": "xxx", "acquired_at": "ISO8601", "token": "uuid-v4"}`（**注意必须包含生成的凭证 token**）。若获取失败则抛出 HTTP 409，前端进入只读模式。后续心跳与释放均必须校验前端传入的 `lock_token` 与 Redis 内 `token` 一致。
* **锁的完整释放策略**：
  * **主动释放**：用户离开编辑页面时，前端在 `useEffect cleanup` 中调用 `POST /locks/release`。后端必须校验 `current_user.user_id == lock_holder.user_id` 且 `token` 一致，否则拒绝释放。
  * **容灾释放**：前端注册 `window.beforeunload` 事件，优先发射最后一次未保存的正文到 `/auto-save`，通过 Fetch keepalive 向 `/locks/release` 发送包含 Header 的释放请求，确保数据不丢失。
  * **提交验证与释放**：后端在 `POST /documents/{doc_id}/submit` 时处理锁逻辑：如果锁 Key 存在，必须校验锁 Key 中的 `user_id` 与当前提交者一致，一致则允许提交并主动删除锁，不一致抛出 409；如果锁 Key 不存在（可能已 TTL 过期），则校验当前提交者必须是该公文的起草人（`creator_id`），若一致仍允许提交（容忍过期），否则抛出 403。
  * **TTL 兜底**：即使上述机制全部失效（如浏览器崩溃），Redis 将在 TTL 到期后自动清除锁。
  * **锁续期心跳**：前端在持有锁期间定期向 `POST /locks/heartbeat` 发送续期请求，重置 TTL。
  * **参数外置配置**：锁的基础 TTL 时长（默认 180 秒）和心跳间隔（默认 90 秒）必须抽取至后端的 `config.py` 环境变量中，以支持按不同业务场景或测试需求灵活调整，并在必要时可对外提供供管理员操作的“强制夺锁”强拆干预 `DELETE /locks/{lock_key}` API，该操作必须写入审计日志，记录管理员身份与被夺锁的牺牲用户，以突破僵死等待。
* **SIP 哈希存证与防篡改稽核**：科长在签批通过时，后端提取终稿内容、时间戳与操作人身份。在生成和校验 SIP 签名之前，必须在 Python 后端强制增加一层文本正则清洗管道，处理全角空格、标点差异和不可见字符。规范化流水线伪代码如下：
  ```python
  def normalize_for_sip(text: str) -> str:
      import unicodedata
      text = unicodedata.normalize('NFKC', text)
      text = text.replace('\r\n', '\n').replace('\r', '\n')  # 换行归一化
      text = text.replace('|', '\\|')                        # 对存证分隔符进行转义
      text = re.sub(r'[^\S\n]+', ' ', text)                  # 将非换行空白符合并为一个空格
      text = re.sub(r' +', ' ', text)                        # 压缩连续空格
      text = text.strip()
      return text
  ```
  做完文本归一化处理后，使用注入的 `SIP_SECRET_KEY` 执行 **HMAC-SHA256** 生成指纹。
  **存证原串公式**：`normalized_content | reviewer_user_id | reviewed_at_iso8601`（`|` 为分隔符）。注意：如果处理后的 `normalized_content` 为空或仅含空白，审批通过时必须拒绝生成 SIP 签名（留空），防止出现无意义的防伪签章。
  系统后续提供 `GET /documents/{doc_id}/verify-sip` 校验接口，计算比对落入 `document_approval_logs` 的哈希值。注意，验证时须先按 `|` 分割字符串，再对内容中的 `\|` 还原回 `|` 以保证验证的一致性。

### 4. 签批驳回与回退流
* **驳回操作**：科长在 `/approval/{doc_id}/review` 接口选择"驳回"时，**必须提供 `rejection_reason`**（后端校验非空），系统执行：
  1. 在 `document_approval_logs` 表插入 `decision_status = REJECTED` 的新记录，含驳回理由。注意：`APPROVED` 的终态记录有且仅有一条，每次提交都会产生新的待审或流转记录。
  2. 在 `nbs_workflow_audit` 审计表注入 `REJECTED(41)` 节点，`action_details` 包含驳回理由 JSON。
  3. 将 `documents.status` 更新为 `REJECTED`。
  4. 通过 SSE 或通知机制向起草人推送驳回消息。
* **回退编辑**：起草人通过 `POST /documents/{doc_id}/revise` 接口将公文状态从 `REJECTED` 回退为 `DRAFTING`（注意：必须严格校验 `status == REJECTED`），审计表注入 `REVISION(42)` 节点，允许重新编辑。**在此事务中，必须同步将 `ai_polished_content` 和 `draft_suggestion` 字段清空以去除可能的旧版本 DIFF 污染**。在完成状态回退的同时，**必须使用 Redis 的 `SET lock:{doc_id} token NX EX 180` 命令原子性地抢占锁**。若抢占失败则抛出 HTTP 409，告知锁已被抢占，不再颁发 Token；若抢占成功，自动为该用户颁发并返回有效的锁 Ticket。`POST /documents/{doc_id}/revise` 接口的响应体结构必须包含：
  ```json
  {
    "doc_id": "uuid",
    "new_status": "DRAFTING",
    "lock_acquired": true,
    "lock_token": "uuid-token",
    "lock_expires_at": "ISO8601"
  }
  ```
  前端收到此响应后，必须将 `lock_token` 注入到 `useLockGuard` 的状态管理中，后续心跳和释放均使用此令牌。前端只需一次请求即可带锁进入工作区，彻底消除抢占真空期缝隙。

## 三、 核心算法设计 (Core Algorithms)

### 1. 混合检索算法 (HRAG: Hybrid Retrieval-Augmented Generation)
单纯的向量检索在面临"查询2024年一季度第一产业总产值"这种强精确数值词时极易失效，必须采用双路召回与融合打分。

> **对 AI Agent 的算法指令**：在 `chat.py` 混合检索算法中，**无论 HNSW 还是 BM25 计算，在最底层 SQL 查询中必须强制 JOIN `knowledge_base_hierarchy` 并共同过滤 `h.is_deleted = FALSE` 与 `knowledge_chunks.is_deleted = FALSE`**，严禁仅依赖单表字段导致目录级联删除穿透幽灵泄露。同时强制叠加底层权限条件过滤：对于 `kb_tier=BASE` 允许全部访问；`DEPT` 限制同科室成员（并在底层查询加上 `knowledge_chunks.dept_id` 匹配或 NULL 的条件以加速鉴权过滤）；`PERSONAL` 仅限 `owner_id = current_user.user_id`。

* **路经 A（语义向量召回）**：
  提取用户 Query 的 Embedding，使用 `pgvector` 的 `<=>` (Cosine Distance) 算子去 `knowledge_chunks` 表中查出 Top-K。
* **路经 B（BM25 全文精准召回）**：
  使用 PostgreSQL 的全文检索函数。为保障良好的中文搜索词法支持，必须为数据库安装 `pg_jieba` 或 `zhparser` 扩展。随后利用 `to_tsquery` 和 `ts_rank_cd` 等函数结合特定的词典进行查询，在同一个切片表中按词频查出 Top-K，保证索引建立阶段与查询阶段中文字典配置的强一致。
* **RRF (Reciprocal Rank Fusion) 重排序融合**：
  计算倒数排名融合得分：`RRF_Score = 1 / (k + Rank_Vector) + 1 / (k + Rank_BM25)`（k 常设为 60）。取融合后的前 3~5 个 Chunk，并在拼接给大模型之前二次遍历排除任何落库时 `security_level='CORE'` 的高绝密召回切片（将其静默废弃并入系统安全审计日志）。增加一个后置校验兜底：若 BM25 路返回候选数为 0 且向量路最高相似度低于阈值（如 0.7），则直接触发“未探明线索”话术，不强行给 LLM 拼接低质量上下文。之后将其拼接为最终的 LLM 提示词上下文。
* **防幻觉系统提示词（System Prompt）装配要求**：
  必须在服务层通过占位符形式植入如下系统角色设定模板，并支持动态配置下发：
  _“你是一个严肃政务问答助手。请严格且只能依据下方【挂载上下文】的数据进行总结回答。若提供的信息不足以响应指令，请直接明示‘未探明对应线索’，严禁依靠自身推理或幻觉编造任何指标流或数据信息。”_

### 2. 知识库切片与多模态解析算法 (Chunking & Parsing)
* **文本降维处理**：对于 `.docx`、`.pdf`，调用 `markitdown` 库提取纯文本。
* **递归分块算法 (AST Semantic Chunking)**：禁止全文直接入库。严禁使用单纯的字符长度暴力切片（如 RecursiveCharacterTextSplitter）。必须采用特定的底层一致性解析器库 `markdown-it-py` 进行基于 AST 的语义层级切割标点：先解析 Markdown/文档结构树，提取标题层级（H1-H6）作为锚点路径。在保证每个逻辑单元完整性的前提下，仅在单元内部超出 800 字符时执行次级切割。确保标题与其从属段落永不分离。
* **表格三维坐标提取 (Tabular RAG)**：如果探测到是 `.xlsx` 等数据表格式，必须抽取出表头，为该表格的每一行数据生成描述摘要，连同原始 JSON 行列数据存入 `metadata_json`。

### 3. GB/T 9704-2023 政务公文物理排版算法
* **词库清洗拦截**：在交由 `python-docx` 渲染前，对传入的 `content` 运行正则拦截机制。屏蔽特定副词（如"隆重召开"、"亲自指示"统一替换为"召开"、"指示"），确保行文符合党政机关行文规范。
* **公文内容排重**：在生成 Word 前，提取正文关键段落哈希，与 `documents` 表内已审批公文（条件：`status = 'APPROVED'` 且同科室）进行相似度比对。若相似度 > 70%，在排版产物中自动标注引用来源或向用户发出雷同预警。（注：与上级下发公文库的跨系统横向排重功能列入二期外部 API 桥接研发计划扩展。）
* **两办规范语料库校验**：增加对“禁止搭配不当”、“句子成分残缺”、“逻辑跳跃”的离线规则扫描（不依赖大模型），在排版前给出语法规范性建议。
* **基于 AST 的结构树遍历**：在物理排版解析阶段保持解析引擎底层依赖一致性，同样使用 `markdown-it-py` 将 Markdown 文本解析为抽象语法树，遍历节点：
  * **一级标题 (##)**：设置字体为"黑体"，三号，无加粗。
  * **二级标题 (###)**：设置字体为"楷体_GB2312"，三号。
  * **正文**：设置字体为"仿宋_GB2312"，三号，首行缩进 2 字符，行距固定值 28 磅。
  * 产物生成在 `/app/data/outputs` 目录下，并返回下载 URL。

## 四、 后端 API 接口地图 (API Endpoint Map)

在 FastAPI 路由器中，按以下模块组织并挂载路由：

| 模块前缀 | 核心接口 | 职责描述 |
| :--- | :--- | :--- |
| **`/api/v1/users`** | `GET /`, `POST /`, `PUT /{id}`, `DELETE /{id}` | 【新增】用户 CRDU 与角色分配（供中枢设置调用）。 |
| **`/api/v1/auth`** | `POST /login`, `GET /me`（返回值结构必须包含 `client_ip` 字段。后端必须配置 FastAPI 使用 `TrustedHostMiddleware` 及前置 Nginx 传递 `X-Forwarded-For` 头以获取真实 IP）, `POST /refresh` | 用户鉴权、JWT 颁发、Refresh Token 静默刷新。 |
| **`/api/v1/documents`**| `GET /?status=&dept_id=&page=`, `POST /init?template_id=`, `GET /{doc_id}`, `DELETE /{doc_id}`, `POST /{doc_id}/auto-save`, `POST /{doc_id}/apply-polish`（接收可选 `final_content`）, `POST /{doc_id}/discard-polish`, `POST /{doc_id}/submit`, `POST /{doc_id}/revise` (返回锁凭证), `GET /{doc_id}/snapshots`, `POST /{doc_id}/snapshots`, `POST /{doc_id}/snapshots/{snapshot_id}/restore`, `GET /{doc_id}/download` | 列表获取必须实现三层权限隔离：管理员可见全局；科长可见本科室所有；科员可见本科室且排除他人 `DRAFTING` 状态草稿（已提交/驳回对科室全员可见，**审批管控台可见性遵循同等 `role_level >= 5` 或 `dept_head_id` 的逻辑**）。接口提供：列表筛选、通过模板 ID 创建、详情、软删除、自动保存、接受/丢弃润色、提交审批、驳回回退、快照查询/手动创建/恢复。排版后 `.docx` 产物安全下载（强校验操作人的 `dept_id` 等于 `documents.dept_id` 且公文未软删。**文件必须通过 `StreamingResponse` 流式返回，绝不暴露真实物理路径**）。 |

## 五、 后端运行安全与物理防线 (Security & Defense)

1. **统一网络路由守卫 (Routing Guard)**：默认允许 `10.132.x.x`，配置 `ALLOWED_SUBNETS` 环境变量支持扩展。
2. **大语言模型 API 限流 (Rate Limiting)**：基于 Redis Token Bucket 算法，对涉及高算力的路由（如 `POST /chat` 和 `POST /tasks/polish`）严格限流（如单用户 5 次/分钟），超出抛出 429 Too Many Requests 以防恶意发包瘫痪 Ollama 节点。
3. **安全双轨化文件回收与去重**：Celery Beat 任务。**关键实现**：使用 Redis 分布式锁确保单实例运行，每次任务限制清理上限（如 1000 个文件），防止堆积。
3. **密码安全**：Argon2id。
4. **SIP 签名与密钥管理**：环境变量注入 `SIP_SECRET_KEY`。签名流程严格遵循归一化流水线。
5. **审计存储事务保护**：审计日志写入必须在主事务 `After-Commit` 钩子内执行，或通过 Celery 异步化，严禁因审计写入失败回滚业务。
6. **协作冲突与静默释放锁**：后端 `/locks/release` 接口应支持标记为 `INACTIVITY` 的释放原因，并写入审计日志以备查验为何锁被系统自动回收。

7. **数据库备份与灾难恢复策略**
   尽管应用层具备快照容灾机制，底层的数据持久化仍需依赖数据库原生备份。服务器必须配置 `pg_dump` 每天定时全量备份与基于 Write-Ahead Logging (WAL) 的归档策略。结合 `pg_basebackup` 确保在发生彻底硬盘损坏或被恶意软件勒索等灾难性故障时，能够执行 Point-in-Time Recovery (PITR) 恢复到具体发生时间点。

## 六、 日志与监控规范 (Logging & Monitoring)

### 1. 结构化日志标准
后端所有模块必须使用 Python `logging` + `structlog` 输出 JSON 格式结构化日志，包含以下必备字段：
* `timestamp`：ISO 8601 格式
* `level`：`INFO` / `WARNING` / `ERROR` / `CRITICAL`
* `module`：来源模块（如 `api.tasks`, `worker.polish`）
* `user_id`：操作人 ID（如有）
* `doc_id` / `task_id`：关联的业务实体 ID（如有）
* `trace_id`：全局请求链路追踪 ID（基于中间件全局拦截器在入口主动生成唯一 `request_id`，并向下贯穿下放至 Celery 的 Task kwargs 及底层日志链中，实现全量生命周期的串联组装关联查找，SSE 推流响应亦需携带此 ID）
* `message`：日志正文

### 2. 关键操作审计日志
以下操作必须输出 `INFO` 级别日志，并同步写入 `nbs_workflow_audit` 表：
* 公文创建/提交/签批/驳回
* 知识库上传/删除/替换
* 用户登录/登出
* 锁的获取/释放/超时/强制催办 (Nudge)

### 3. 异常告警与重试退避机制
* Celery Worker 进程退出或长时间无响应时，通过标准错误输出 `CRITICAL` 级别日志。
* **任务重试联动 (Retry Bound)**：在 Celery 任务装饰器显式设定 `max_retries=3` 和 `autoretry_for=(Exception,)`。任务内部捕获异常后，首要操作是更新 `async_tasks.retry_count += 1`。若 `retry_count <= 3`，调用 `self.retry()` 触发重试并**结束当前函数执行**；若 `retry_count > 3`，将状态置为 `FAILED`，存储错误堆栈，并不再抛出异常（阻断 Celery 内部重试接管）。这确保计数与重试严格一致。
* 环境参数必须在 `config.py` 中明确落定并暴露：Redis 连接超时降级为 5s、数据读写超时压控为 3s；Ollama HTTP 请求阻塞阀定格 120s。且 Celery 异步任务池必须硬防死锁滞留，装配 `soft_time_limit=600` 软熔断及 `time_limit=900` 线程斩断强裁决策机制。并将安全归档冷存储根路径规划给 `ARCHIVE_ROOT`。
* 建议对接日志收集系统（如 ELK 或 Loki + Grafana），并配置以下告警规则：
  * 5 分钟内 `ERROR` 日志超过 10 条
  * `async_tasks` 表中 `FAILED` 任务占比超过 20%
  * Redis 连接断开超过 30 秒
