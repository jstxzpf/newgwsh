# 泰兴市国家统计局公文处理系统 V3.0 - 整体设计方案

## 一、 系统功能定位与核心模块

本系统旨在为国家统计局县级基层调查队打造集数据合规、物理排重、灾难自愈、算力协同于一体的智慧公文处理中枢。系统深度融合了统计调查业务流程、政务公文标准排版规范以及大语言模型技术。

### 2. 统计知识资产管理
* **多模态数据摄入**：支持单文件与多层级目录的并发上传**（需明确支持前端压缩包/多文件 API 传输，服务端由 Celery 分发解压与入库，避免阻塞主线程）**，内置 LibreOffice 引擎兼容旧版 `.doc`/`.xls` 格式，并由 MarkItDown 提取富文本内容**（需外挂后处理清洗节点以应对复杂 PDF/Word 解析后的 AST 失真）**。
* **严密的权限分级**：数据划分为基础库、科室库与个人库，并辅以"核心/重要/一般"的安全等级标签，防止核心统计数据发生跨权越权泄露。
* **版本化管理**：支持知识资产的替换上传，系统自动递增版本号、重新切片并清理旧版本向量索引，同时保留历史版本的审计记录。

### 3. HRAG 穿透式智能问答
* **定向范围挂载**：用户可在前端工作区或问答界面，定向勾选特定的统计目录或台账文件，缩小 AI 检索范围。
* **防幻觉机制**：系统层级注入防幻觉安全契约，强制 AI 仅从挂载的上下文提取数据，遇盲区必须回答"未探明线索"。

### 4. 全局任务与合规审计
* **异步流水线与全局任务守望者**：所有高耗时的 AI 润色、排版及解析任务均由 Celery 后台执行。前端通过 `GlobalTaskWatcher` 全局组件管理 SSE 隧道，配合短期加密 Ticket 提供无阻塞进度推送。用户可“发后即忘”，在任务中心（`/tasks`）或通过全局通知卡片实时获取状态，支持任务失败热重置。
* **核心存储与容灾**：采用“云端优先”存储策略。公文、快照、资产切片统一存储。**快照系统实现“无感存证”，在用户执行“接受润色”等覆盖操作前自动触发备份**。客户端仅保留 2 条本地草稿用于极速容灾。系统监控 `pg_dump` 与 WAL 归档进程，遇错立即触发多端预警。
* **安全审计与 SIP 存证**：引入基于 **HMAC-SHA256** 的 SIP 溯源技术。在签批环节，系统执行标准化的文本归一化流水线（NFKC + 换行压缩 + 空白归一），利用 `SIP_SECRET_KEY` 计算不可篡改的存证指纹。审计日志与业务逻辑解耦，实现从落库、查封到鉴伪的全维闭环。

## 二、 技术框架底座



系统采用严格的前后端分离架构与领域驱动设计 (DDD) 理念，确保高并发下的稳定性与可扩展性。

* **前端交互层 (Frontend)**
    * **核心框架**：React 19, TypeScript, Vite。
    * **UI 引擎与状态**：Ant Design v6 构建界面，Zustand 负责跨组件状态与持久化管理。
    * **通信与安全**：封装带有 JWT 认证拦截的 Axios 实例，使用 `sse.js` 处理流式连接，全局覆盖无感防泄密水印。
* **后端业务层 (Backend)**
    * **Web 框架**：FastAPI (Python 3.10+)，提供极高吞吐量的异步路由网关。
    * **任务编排**：Celery 搭配 Redis Broker，隔离并调度重型 I/O 与 AI 推理任务。
    * **防呆与安全路由**：内置流量路由守卫（Routing Guard），阻断旁路网关非法访问，仅放行 `10.132.60.1` 及合法内网流量。
* **持久与缓存层 (Data & Middleware)**
    * **关系型数据与向量**：PostgreSQL 15，并依托 `pgvector` 插件实现高维（1024 维）向量检索，适配 bge-m3 语义引擎。
    * **ORM 会话隔离**：API 层采用 `asyncpg` 驱动的异步会话，Celery 后台任务采用 `psycopg2` 驱动的同步会话，物理规避事件循环阻塞。
...
    * **后端逻辑与监控**：内置流量路由守卫与资源嗅探探针，实时采集 CPU、内存及 AI 节点连通性。
    * **高速缓存与分布式锁**：Redis 5.0，不仅用作消息队列，同时支撑 Redlock 悲观锁机制以防止多用户覆盖写。
* **AI 算力层 (AI Engine)**
    * **推理服务**：跨节点穿透至 `10.132.60.133` 的 Ollama 服务，驱动百亿参数级的 `gemma4:e4b` 重型模型执行自然语言处理任务。

## 三、 用户核心工作流程

### 1. 公文全生命周期流转 (Document Lifecycle)

> 核心状态机：`DRAFTING → SUBMITTED → APPROVED (终态)` 或 `SUBMITTED → REJECTED → DRAFTING (回退修改)`

1.  **新建起草**：用户点击“起草新公文”，后端通过 `POST /api/v1/documents/init` 分配全局唯一 `doc_id`，初始化 `DRAFTING(10)` 状态。
2.  **上下文挂载**：左侧“虚拟文档树”支持勾选台账。勾选时前端自动捕获 `context_snapshot_version` (目录树当前时间轴锚点)。
3.  **并发锁控与心跳**：前端通过 **Web Worker** 维持心跳。后端响应中下发 `next_suggested_heartbeat` 校准心跳精度，并返回 `lock_ttl_remaining`。
4.  **自动保存与安全网**：每 60 秒自动保存。前端建立失败计数器，连续 3 次失败则强制报警并阻断操作，防止静默丢数。
3.  **撰写与自动保存**：在 A4 画板完成初稿。后端每 60 秒触发 `POST /api/v1/documents/{doc_id}/auto-save`。系统通过 Redis 悲观锁防止多用户冲突，锁具备心跳续期机制。
4.  **智能润色（异步任务）**：点击“AI 智能润色”，请求 `POST /api/v1/tasks/polish` 后立即返回 `task_id`。用户可离开当前页处理其他事务。画板显示骨架屏动画：“AI 正在研读挂载台账...”。
5.  **DIFF 模式与建议合并**：任务完成后，全局通知卡片提醒用户。进入工作区自动切为 `DIFF` 模式。用户可在右栏“建议稿”进行二次修改，该修改存入 `draft_suggestion`。点击“接受并合并”后，系统备份原稿至快照并覆写正文。
6.  **提交审批**：点击“提交审批”，系统自动触发一次 `auto-save`。后端校验 `DRAFTING` 状态并释放编辑锁。状态变更为 `SUBMITTED(30)`。
7.  **签批通过（终态）**：负责人执行 `APPROVED(40)` 操作。后端对终稿执行 **NFKC 归一化流水线**，利用 `SIP_SECRET_KEY` 计算 HMAC-SHA256 存证指纹落库。公文转为终态，禁止编辑。
8.  **签批驳回与原子回退**：负责人选择驳回并填写理由（`REJECTED(41)`）。起草人点击“前往修改”，触发 `POST /api/v1/documents/{doc_id}/revise`。后端在同一事务内完成：状态回退、**清空旧 AI 缓存**、以及 **原子性抢占 Redis 编辑锁**。抢占成功后下发锁凭证，用户带锁接管编辑。
9.  **国标排版下载（全自动触发）**：审批通过时，后端在同一事务的 `After-Commit` 钩子中**自动派发** `task_type=FORMAT` 的 Celery 异步排版任务（方案A），无需用户手动操作。Worker 按照 GB/T 9704-2023 规范（黑体 H1、楷体 H2、仿宋正文、28 磅行距等）生成 `.docx`。前端通过 `GlobalTaskWatcher` 接收 `FORMAT` 任务完成通知卡片，用户点击"立即下载"后通过 `StreamingResponse` 安全获取产物。

### 2. 统计资产管理与消费流 (Knowledge Asset Management)

为实现资产的“上传便捷”与“层次分明”，系统构建了端到端的知识库闭环：

1.  **层次化目录体系 (Tiers & Hierarchy)**：
    *   **三级分仓**：系统划分为全局基础库（`BASE`，全员只读）、科室共享库（`DEPT`，本科室只读）、个人沙箱库（`PERSONAL`，私人专用）。权限严格由角色等级（`role_level`）控制。
    *   **多级虚拟树**：通过 `knowledge_base_hierarchy` 支持无限级目录结构。用户可创建目录（`DIRECTORY`）或上传文件（`FILE`）。
2.  **便捷上传与物理去重 (Ingestion)**：
    *   **多样化上传**：支持单文件、多文件、本地文件夹直接拖拽，以及 `.zip/.tar.gz` 压缩包上传（后端自动解压并还原目录结构）。
    *   **物理文件去重**：采用 SHA-256 计算物理哈希。逻辑节点创建前优先检索 `knowledge_physical_files`，实现“一处存储，多处引用”，极大节省存储空间并加速解析。
3.  **资产安全标签 (Security)**：
    *   每个资产必须设定安全等级（一般、重要、核心）。核心资产（`CORE`）将被标记为敏感，禁止作为 RAG 上下文推送给外部 AI 模型。
4.  **异步解析与切片复用 (Parsing & Reuse)**：
    *   **状态接力**：文件上传后进入 `UPLOADED` 状态。若该物理文件已有有效切片，逻辑节点直接复用并转为 `READY`。
    *   **AST 语义切片**：新文件由 Celery 执行 `PARSE` 任务。利用 `markitdown` 提取内容，并基于 `markdown-it-py` 进行 AST 语义分块。
    *   **Tabular RAG**：针对 `.xlsx/.csv`，系统自动识别表头并为每行生成摘要描述，连同行列 JSON 存入切片元数据。
    *   **标题路径注入**：每个切片强制注入完整标题路径（如“全局库 > 统计法 > 第一章”），确保召回时具备完整上下文语义。
5.  **版本更新与资产更新 (Versioning)**：
    *   支持“替换上传”。若哈希变化，系统自动建立新版本，并将旧版本切片标记为软删除（`is_deleted=True`）。
6.  **资产消费与全链路防护 (Consumption)**：
    *   **穿透消费**：用户勾选目录或文件，系统递归检索其下所有有效切片，结合权限过滤后送入大模型。
    *   **软删除清理**：删除目录时，通过 `WITH RECURSIVE` CTE 实现级联软删除，强制将对应切片的 `embedding` 置为 `NULL`，从向量检索中彻底移除。
